# Claude Codeで強化学習の実験サイクルを半自動化した話

## はじめに

強化学習（RL）でロボットの歩容を獲得する研究では，「報酬関数を少し変えて訓練し直す」というイテレーションを高速に回すことが成果に直結します．しかし実際にやってみると，各バージョンで必要な作業は意外と多いのです．

1. （あれば）前バージョンのレポートを読み，次に試すべき変更を確認する
2. 訓練スクリプトをコピーして修正する
3. コード品質チェック（lint・型チェック）を通す
4. 訓練を実行して完了を待つ
5. 評価スクリプトを複数実行してデータを収集する
6. 定量分析を行い，前バージョンと比較する
7. 考察を書き，次バージョンの改善案を提案する
8. レポートにまとめる
9. 最初に戻る

これを毎バージョンやるのは正直しんどいですよね．

私はこのワークフローをClaude Codeで半自動化することで，3つの実験シリーズにわたって計80バージョン以上のイテレーションを回しました．人間による介入が必要だったのは，最終的に **「得られたポリシーを物理シミュレータで動かして目視評価すること」** と **「次の実験セッションを開始するコマンドを打つこと」** だけです．

本記事では，このワークフローを実現するためにClaude Codeの各機能をどう組み合わせたかを紹介します．

## 前提条件

- Claude Codeの基本操作を知っている方が対象です（`CLAUDE.md` の存在は知っている程度でOK）
- 強化学習そのものの詳細は本記事のスコープ外です．「訓練スクリプトを実行→評価→分析→改善」のサイクルがある，という程度の理解で十分です

## 全体像 — 何が自動化され，何が残るのか

まず全体像を示します．以下の図は1バージョンの実験サイクル全体を表しています．

![全体ワークフロー](./art003_introduce_dev_with_ai_workflow.drawio.svg)

ワークフローは4つのフェーズからなります．

| フェーズ | 内容 | 自動化状況 |
|---------|------|-----------|
| Phase 1: 準備 | 前版レポート読込→スクリプト生成→lint→レポート骨格作成 | 完全自動 |
| Phase 2: 訓練 | バックグラウンド実行＋tmux監視ペイン作成 | 半自動（実行は自動，完了待ちは人間） |
| Phase 3: 分析 | データ収集→定量分析→独立考察→レポート執筆 | ほぼ完全自動 |
| Phase 4: 完了 | 結果サマリ＋次バージョン提案 | 完全自動 |

**人間がやること** は以下の2つだけです．

1. **目視評価**: 訓練済みポリシーを物理シミュレータで動かし，「歩容が安定しているか」「左右対称か」などを目で確認する．所感は一言テキストで渡す（例: 「右足の接地が不安定，やや左に傾く」）
2. **セッション開始**: 次のバージョンを作るコマンドを実行する（例: `/exp008-new-version 25`）

つまり，ターミナルで1つのスラッシュコマンドを打てば，Phase 1〜4が順次実行されます．訓練が終わったら目視評価して所感を渡し，また次のコマンドを打つ——この繰り返しです．

## Claude Codeの構成要素と全体設計

この自動化を支えるClaude Codeの構成要素は以下の5つです．

![構成要素の関係](./art003_introduce_dev_with_ai_architecture.drawio.svg)

| 構成要素 | ファイル | 役割 |
|---------|---------|------|
| **AGENTS.md** | `AGENTS.md` | プロジェクト全体のルール（コード品質，ドキュメント形式，作業手順） |
| **Skills** | `.claude/skills/exp008-new-version/SKILL.md` 等 | 実験ごとのE2Eワークフロー定義．スラッシュコマンドとして呼び出せる |
| **Hooks** | `.claude/hooks/lint-check.sh` | ファイル編集時に自動発火するコード品質チェック |
| **Settings** | `.claude/settings.json` | 許可コマンド，環境変数，hooks設定 |
| **Rules** | `doc/experiments/exp{NNN}_rules.md` | 実験固有のルール・制約（コピー原則，1変更1検証原則など） |

これらは独立した部品ですが，組み合わせることで「ルール駆動の半自動ワークフロー」を構成しています．以降のセクションでは，各構成要素の詳細と設計の勘所を解説します．

## Skills — ワークフローをスラッシュコマンドにする

Skillsは，Claude Codeの最も強力な機能の一つです．`.claude/skills/` ディレクトリにMarkdownファイルを配置するだけで，カスタムのスラッシュコマンドとして呼び出せるようになります．

### Skillの構造

私のプロジェクトでは，1つの実験シリーズにつき1つのスキルディレクトリを作成しています．

```
.claude/skills/
├── exp008-new-version/
│   ├── SKILL.md              # メインの定義ファイル（スラッシュコマンドの入口）
│   ├── phase1-prepare.md     # Phase 1のチェックリスト
│   ├── phase3-analyze.md     # Phase 3のAgent Teams構成
│   └── report-template.md    # レポートの骨格テンプレート
├── exp009-collect/
│   └── SKILL.md              # データ収集専用スキル
└── exp009-evaluate/
    └── SKILL.md              # 分析・レポート更新専用スキル
```

`SKILL.md` のフロントマターでスキルのメタデータを定義します．

```yaml
---
name: exp008-new-version
description: Create a new exp008 training version with full E2E workflow.
argument-hint: "[version-number]"
---
```

`argument-hint` を設定しておくと，`/exp008-new-version 25` のように引数付きで呼び出せます．スキル本文中では `$ARGUMENTS` で引数を参照できます．

### SKILL.md に書くこと

SKILL.md本文には，フェーズごとの具体的な手順を記述します．ポイントは **「何をすべきか」だけでなく「どうやるか」まで具体的に書く** ことです．

```markdown
## Phase 1: 準備

1. 前バージョンのレポート `exp008_report_v{N-1}.md` を読み込み、
   「次バージョンへの提案」を確認
2. トレーニングスクリプトをコピー原則に従い作成
   - `cp biped_walking/train/droid_train_narrow_v{N-1}.py
        biped_walking/train/droid_train_narrow_v{N}.py`
3. 必要に応じて `droid_env_unitree.py` に報酬関数を追加
4. lint・型チェック（`AGENTS.md` コード品質ルール参照）
5. レポートスケルトンを report-template.md に従い作成
```

このレベルの具体性があれば，Claude Codeは迷わず実行できます．

### スキル分割の設計判断

実験シリーズexp009では，1つのスキルを3つに分割しました．

| スキル | 用途 | 分割理由 |
|-------|------|---------|
| `/exp009-new-version N` | Phase 1〜2（準備＋訓練） | 訓練完了まで人間が待つ必要がある |
| `/exp009-collect N` | Phase 3前半（データ収集） | 目視評価の**前に**定量データを揃えたい |
| `/exp009-evaluate N "所感"` | Phase 3後半〜4（分析＋レポート） | 目視評価の所感を**引数で**受け取る |

この分割により，ワークフローの間に **人間の判断ポイント**（目視評価）を自然に挟み込めます．「全部自動化すれば良い」わけではなく，人間が介入すべきポイントをスキルの境界として設計するのが重要です．

## Hooks — 編集のたびに自動でコード品質を担保

Claude Codeにコードを書かせると，たまにlintエラーや型エラーが混入します．Hooksを使えば，ファイル編集のたびに自動でチェックを走らせ，その場で修正させることができます．

### 設定方法

`.claude/settings.json` に `PostToolUse` hookを設定します．

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/lint-check.sh",
            "timeout": 60
          }
        ]
      }
    ]
  }
}
```

`matcher: "Edit|Write"` により，EditツールまたはWriteツールが実行されるたびに `lint-check.sh` が発火します．

### lint-check.sh の設計

このスクリプトは以下の判断ロジックを持っています．

1. **対象判定**: Pythonファイルかつ `rl_ws/` 以下のファイルのみ検査（submoduleやvenvは除外）
2. **Ruff lint**: 問題があれば自動修正を試み，手動修正が必要な場合はフィードバック
3. **Mypy型チェック**: 型エラーをフィードバック

```bash
# 対象ファイルのフィルタリング（抜粋）
if [[ ! "$FILE_PATH" == *.py ]]; then
    exit 0
fi
if [[ "$FILE_PATH" == *"genesis_official"* ]] || \
   [[ "$FILE_PATH" == *".venv"* ]]; then
    exit 0
fi
```

hookのフィードバックはClaude Codeにそのまま渡されるため，エラーが検出されればClaude Codeが自律的に修正します．人間がレビューする前にコード品質が担保される仕組みです．

## Rules — AIの振る舞いの質を決める鍵

自動化の品質を左右するのは，実はSkillsやHooksではなく **ルールドキュメント** です．AIに「何をするか」だけでなく「何をしてはいけないか」「なぜそうするのか」を明示することで，振る舞いの質が大きく変わります．

### AGENTS.md — プロジェクト全体のルール

`AGENTS.md`（または `CLAUDE.md`）にはプロジェクト全体で守るべきルールを記載します．私のプロジェクトでは以下のような内容を含めています．

- リポジトリ構成と各ディレクトリの役割
- 作業ディレクトリのルール（`cd rl_ws` してからコマンド実行）
- コード品質基準（PEP-8，行長120文字，型ヒント）
- ドキュメント作成ルール（日本語，`.drawio.svg` 形式の図，アスキーアート禁止）
- 実験レポートの責務分離ルール

### 実験ルールファイル — ドメイン固有の制約

汎用的なルールだけでは不十分です．各実験シリーズ固有の制約は `exp{NNN}_rules.md` に記載し，Skillsから参照させます．

私のプロジェクトで特に効果が大きかったルールを3つ紹介します．

#### 1. コピー原則

```markdown
新バージョンのスクリプトは必ず前バージョンからコピーして作成し，
ゼロから書き直してはならない．
```

RL訓練スクリプトには多くの隠れた依存関係（関節インデックス，観測ベクトルの次元数など）があります．ゼロから書くとバグの温床になるため，「コピーして最小限の変更だけ加える」原則を明記しました．

#### 2. 1変更1検証原則

```markdown
1バージョンで変更する項目は1〜2個に限定する．
複数の変更を同時に行うと，結果の因果関係が追えなくなる．
```

AIに任せると「せっかくだからここも改善しよう」と複数の変更を一度に入れがちです．この原則を明記することで，変更の粒度を適切に保てます．

#### 3. 独立考察（Step D）のスキップ禁止

```markdown
Step D（独立考察）は省略禁止．データ収集から直接レポート執筆に進むことは禁止．
考察すべき項目:
1. 動作悪化メカニズムの因果関係の特定
2. 報酬設計全体の体系的整理
3. 複数改善案の設計と評価
4. 推奨案の決定
```

データを集めてそのままレポートに貼り付けるだけでは，「なぜこうなったのか」「次に何をすべきか」の考察が浅くなります．独立した考察ステップを**スキップ禁止**として明記することで，分析の質を担保しています．

### ルール記述の勘所

経験上，以下の指針が効果的でした．

| 悪い例 | 良い例 |
|--------|--------|
| 「コードの品質を保つこと」 | 「Edit/Write後にruff checkとmypyを実行し，エラーがあれば修正すること」 |
| 「適切に分析すること」 | 「Step D（独立考察）は省略禁止．データ収集から直接レポート執筆に進むことは禁止」 |
| 「前のバージョンを参考に」 | 「`cp droid_train_v{N-1}.py droid_train_v{N}.py` でコピーしてから最小限の変更のみ行う」 |

曖昧な目標ではなく，具体的な手順と禁止事項を書くことがポイントです．

## Agent Teams — 分析フェーズの並列化

実験が進むにつれ，分析フェーズで実行すべきコマンドが増えていきました．7〜8個の独立した評価・分析コマンドを逐次実行すると時間がかかります．そこで，Claude CodeのAgent Teams機能を活用して並列化しました．

### Agent Teamsの有効化

`settings.json` の環境変数で有効にします．

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### exp008での活用: 並列データ収集

exp008のPhase 3では，Agent Teamsを使って以下のように並列化しています．

```
リーダーエージェント
    ├─ data-collector（タスク1-7: 各種評価・分析コマンドを並列実行）
    │   ├─ 基本評価
    │   ├─ CSV評価（V{N} と V{N-1}）
    │   ├─ 訓練ログ分析
    │   ├─ 報酬コンポーネント分析
    │   └─ 横方向揺れ・振動分析
    │
    ├─ survey-analyst（タスク9: 先行研究との照合）
    │
    └─ リーダー自身（タスク8,10,11: 定量分析→独立考察→レポート執筆）
```

独立したデータ収集タスクを `data-collector` エージェントに委譲し，リーダーは収集完了後に定量分析と考察に集中します．

### exp009での活用: 二者対話による多角的分析

exp009ではさらに進んで，**2つの専門視点による対話形式の分析**を導入しました．

| ラウンド | 役割 | 内容 |
|---------|------|------|
| Round 1 | DA（データ分析） | 定量データの精密な読み解き，異常値・変化の特定 |
| Round 2 | Lead（物理解釈） | DAの分析結果を物理的メカニズムで解釈，仮説構築 |
| Round 3 | DA | Leadの仮説を検証するための追加データ提示 |
| Round 4 | Lead | 統合考察，改善案の提示 |

各ラウンドは独立したTaskとして実行され，リーダーが結果を次のラウンドに受け渡します．これにより，単一視点では見落としがちな洞察が得られました．

たとえば，あるバージョンで「報酬ベースの対称化が構造的に失敗する」という知見は，DAが定量的な非対称度の持続を報告し，Leadがそれを物理学の自発的対称性の破れ（Spontaneous Symmetry Breaking）として解釈したことで初めて得られたものです．

## Settings — 許可コマンドと安全策

`settings.json` は，Claude Codeの振る舞いを制御する設定ファイルです．特に重要なのが `permissions.allow` による許可コマンドの管理です．

### 許可コマンドの設計

自動実行を許可するコマンドを明示的にリストアップします．

```json
{
  "permissions": {
    "allow": [
      "Bash(cp biped_walking/train/droid_train_*.py biped_walking/train/droid_train_*.py)",
      "Bash(uv run ruff *)",
      "Bash(uv run mypy *)",
      "Bash(uv run python biped_walking/biped_eval.py *)",
      "Bash(uv run python biped_walking/train/droid_train_narrow_v*)",
      "Bash(uv run python scripts/analyze_training_log.py *)",
      "Bash(uv run python scripts/show_reward_components.py *)",
      "Bash(uv run python scripts/analyze_eval_csv.py *)",
      "Bash(tmux *)"
    ]
  }
}
```

ポイントは **ワイルドカード `*` の使い方** です．`droid_train_narrow_v*` のようにバージョン番号部分だけをワイルドカードにすることで，任意のバージョンの訓練を自動実行しつつ，想定外のスクリプト実行は防止できます．

### `&&` 禁止の理由

`AGENTS.md`には「`cd` と後続コマンドを `&&` で連結してはならない」というルールがあります．これはClaude Codeの仕様に起因します．

```bash
# NG: allowパターンにマッチしない
cd rl_ws && uv run python scripts/analyze_training_log.py ...

# OK: 別々のBash呼び出しに分ける（cwdは保持される）
cd rl_ws        # Bash呼び出し1
uv run python scripts/analyze_training_log.py ...  # Bash呼び出し2
```

`settings.json` の `allow` パターンはコマンド文字列の**先頭からマッチ**するため，`cd rl_ws && uv run ...` は `uv run ...` のパターンにマッチしません．この制約を知らないとスキルが意図通りに動かないため，ルールとして明記しています．

### 環境変数の活用

```json
{
  "env": {
    "PYTHONUNBUFFERED": "1"
  }
}
```

`PYTHONUNBUFFERED=1` を設定することで，訓練ログがバッファリングされずにリアルタイムで出力されます．これにより，バックグラウンド実行中の進捗をtmuxで監視できます．

## まとめ

本記事で紹介したClaude Code構成要素の役割を改めて整理します．

| 構成要素 | 役割 | 設計の勘所 |
|---------|------|-----------|
| **AGENTS.md** | プロジェクト全体のルール | 汎用的な規約を集約．具体的な手順と禁止事項を明記 |
| **Skills** | E2Eワークフローの定義 | 人間の判断ポイントでスキルを分割．手順は具体的に |
| **Hooks** | ファイル編集時の自動チェック | 対象ファイルのフィルタリングが重要．外部ライブラリは除外 |
| **Rules** | 実験固有の制約 | 「曖昧な目標」ではなく「具体的な禁止事項」を書く |
| **Settings** | 許可コマンド・環境変数 | ワイルドカードで柔軟に，かつ安全に |
| **Agent Teams** | 分析の並列化・多角化 | 独立タスクの並列実行＋専門視点の対話 |

これらの構成要素は一度に全部導入する必要はありません．段階的に導入することをおすすめします．

1. **まず Hooks**: ファイル編集のたびにlintを走らせるだけで，コード品質が安定します
2. **次に AGENTS.md**: プロジェクト固有のルールを明記して，AIの振る舞いの質を上げます
3. **そして Skills**: 繰り返しの作業をスラッシュコマンド化して，E2Eの自動化を実現します
4. **最後に Agent Teams**: 分析の並列化や多角的な考察が必要になったら導入します

重要なのは，**「AIに全部任せる」のではなく「人間が判断すべきポイントを見極めてワークフローを設計する」** という姿勢です．本プロジェクトでは，物理シミュレータでの目視評価という「AIには（まだ）難しい判断」を人間が担い，それ以外の定型作業をAIに委ねることで，高速なイテレーションを実現しました．

## おわりに

本記事では，Claude Codeの各構成要素（Skills，Hooks，Settings，AGENTS.md，Agent Teams）を組み合わせて，強化学習の実験サイクルを半自動化した事例を紹介しました．本記事の内容はexp007・exp008・exp009の実験成果に基づいています．

Claude Codeは汎用的なCoding Agentですが，ルールとワークフローを適切に設計すれば，特定ドメインの反復作業を高い品質で自動化できます．皆さんのプロジェクトでも，まずはHooksから試してみてはいかがでしょうか．
