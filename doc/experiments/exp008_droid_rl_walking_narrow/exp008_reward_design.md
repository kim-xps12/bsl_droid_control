# EXP008: 報酬設計原則

exp008ではexp007で確立された報酬設計を継承する。Unitree RL Gymの報酬設計に基づく理論的背景は [exp007_unitree_rl_gym_survey.md](../exp007_droid_rl_walking_ref_unitree/exp007_unitree_rl_gym_survey.md) を参照。

exp007の40バージョンにわたる知見の総括は [exp007_summary_for_exp008.md](../exp007_droid_rl_walking_ref_unitree/exp007_summary_for_exp008.md) を参照。

## 1. 報酬項目追加に関する厳格ルール（exp007 V10教訓に基づく）

**exp007 V1-V10の試行錯誤から得られた重要な教訓**を今後の実験で必ず遵守すること。

### 報酬項目の追加は最終手段

```
❌ 問題発生 → 新しい報酬項目を追加
✅ 問題発生 → 既存項目の重み調整 → 効果なし → 項目の削除を検討 → 最後に追加を検討
```

**根拠**：
- exp007 V3-V4（16-17項目）で0.15-0.19 m/sを達成
- exp007 V7-V10（22項目）で不安定化、最終的に静止ポリシー（0.006 m/s）
- Unitree G1/H1は13-14項目で実機歩行を達成
- **項目を増やすほど良くなるわけではない。むしろ項目間の相互作用が予測困難になる。**

### 新規項目追加前の必須チェックリスト

新しい報酬項目を追加する前に、以下を**全て**確認すること：

- [ ] **既存項目の重み調整で対応できないか？**
- [ ] **不要な項目を削除して簡素化できないか？**
- [ ] **Unitree RL Gymに同等の項目が存在するか？**（存在しない場合は慎重に）
- [ ] **exp007 V3-V4（動作成功時）に存在した項目か？**
- [ ] **追加による副作用（ペナルティ累積、相互干渉）を十分に検討したか？**
- [ ] **一度に追加するのは1項目のみか？**

### 避けるべきパターン（exp007で実証済み）

| パターン | 問題 | 実例 |
|---------|------|------|
| 複数項目の同時追加/強化 | 相互作用が予測困難 | V10: 5項目同時変更 → 静止ポリシー |
| 対称性報酬（symmetry）の強化 | hip_pitch同期誘発 | V9: symmetry 1.0 → 相関+0.605に悪化 |
| ペナルティの累積強化 | 動くこと自体がペナルティに | V7→V10でペナルティ倍増 → 静止 |
| 複雑な交互歩行報酬 | 意図しない効果 | alternating_gait/hip_pitch_antiphase → 効果なし |
| hip_pos compound penalty | hip_yaw+hip_roll同時制約 | V37-V40: 独立制御不可 |
| symmetry_range | 振幅縮小(race to bottom) | V33: 等化ではなく全体縮小 |

### 推奨するアプローチ

1. **シンプルさを維持**: 15-17項目を上限とする
2. **実績ベース**: exp007 V39、Unitreeの設定を参照
3. **1変更1検証**: 一度に変更するのは1-2項目まで
4. **削除優先**: 追加より削除を優先的に検討
5. **ペナルティ控えめ**: 主報酬を上回らないよう設定

### 問題別の対処指針

| 問題 | 推奨アプローチ | 避けるべきアプローチ |
|------|--------------|-------------------|
| 交互歩行が不十分 | single_foot_contactの重み調整 | symmetry/alternating_gait追加 |
| 足上げ不足 | swing_height_target調整 | feet_swing_height強化 |
| Yawドリフト | tracking_ang_vel調整、hip_pos維持 | hip_pos完全除去（V40教訓） |
| 動作が滑らかでない | action_rate微調整 | 複数ペナルティ強化 |
| 静止ポリシー | ペナルティ緩和、velocity_deficit確認 | 新規報酬追加 |
| 横揺れ（lateral sway） | ankle_roll追加、ang_vel_xy調整 | hip_pos単独強化（構造欠陥） |

## 2. 報酬項目数の目安

| 項目数 | 評価 | 根拠 |
|-------|------|------|
| 13-14 | ✅ Unitree推奨 | G1/H1で実機歩行達成 |
| 15-17 | ✅ BSL-Droid推奨 | exp007 V3-V4で0.15-0.19 m/s達成 |
| 18-20 | ⚠️ 注意 | 相互作用リスク増加 |
| 21以上 | ❌ 非推奨 | exp007 V7-V10で不安定化実証済み |

## 3. 階層的報酬設計の原則

報酬スケールは以下の階層構造を遵守すること：

1. **主報酬**（最優先）：tracking_lin_vel（1.0-1.5）> tracking_ang_vel（0.5）
2. **歩行品質報酬**（次優先）：feet_air_time, contact, single_foot_contact
3. **補助報酬**（低優先）：alive, その他
4. **ペナルティ**（抑制のみ）：**絶対値の合計が主報酬を上回らないこと**

**重要**: ペナルティの累積が主報酬を上回ると、「動かない」が最適解になる（exp007 V10で実証）。

## 4. exp008固有の考慮事項

### 4.1 胴体幅縮小による影響

exp008では胴体幅を75%に縮小（0.18→0.135m）、脚間隔も同比率で縮小（0.10→0.075m）。これにより:

- **支持基底面の狭小化**: 横方向バランスが厳しくなる可能性。ankle_roll/ang_vel_xy による制御が重要
- **回転慣性の低下**: izz低下（0.04→0.029）によりYaw制御が容易になる可能性
- **最適報酬パラメータの変化**: exp007で確立した重みが新URDFでは最適でない可能性。V1の結果を踏まえて調整

## 5. トラブルシューティング

### 5.1 「動かない」ポリシーが学習された（静止ポリシー問題）

**これはexp007 V1-V10で繰り返し発生した最も深刻な問題である。**

#### 原因候補（優先度順）

1. **ペナルティの累積効果**（最も可能性が高い）
   - V10で実証: 5項目のペナルティを同時強化 → 静止ポリシー
   - 動くこと自体が大きなペナルティを受ける状態

2. **報酬項目の相互干渉**
   - symmetry報酬がhip_pitch同期を誘発（V9で実証）
   - 複雑な報酬設計が意図しない効果を生む

3. **alive報酬が相対的に高い**
   - Unitreeの0.15は大きすぎる場合がある
   - 動かずに生存するだけで報酬を獲得

4. **速度目標が低すぎる**
   - 低速目標では動く動機が弱い
   - tracking_lin_velの勾配が緩くなる

#### 対処（優先度順）

1. **ペナルティを緩和する**（最優先）
   - exp007 V7レベルに戻す: ang_vel_xy=-0.05, orientation=-0.5
   - 複数項目を一度に強化した場合は全て戻す

2. **不要な報酬項目を削除する**
   - symmetry, alternating_gait, feet_stumble等を削除
   - 15-17項目に簡素化

3. **velocity_deficitを確認**
   - 速度未達ペナルティが有効か確認
   - V6で効果実証済み: -2.0

4. **動作していたバージョンに回帰**
   - exp007 V39またはexp008の直近動作バージョンの設定に戻す
   - そこから最小限の変更で再試行

#### チェックポイント

```
報酬最大化 ≠ 歩行品質最大化
```

exp007 V10は最終報酬98.12（V7の73.44より高い）だが、X速度は0.006 m/s（静止）。
報酬値ではなく、**実際の動作**で評価すること。

### 5.2 接地フェーズ検出がうまくいかない

原因候補:
1. contact_threshold（接触力閾値）が不適切
2. 足先リンクのインデックスが誤っている

対処:
1. 接触力の値をログ出力して確認
2. feet_indicesの設定を確認

### 5.3 feet_swing_heightペナルティが強すぎる

原因候補:
- Unitreeのスケール（-20.0）がBSL-Droidには過剰

対処:
- スケールを1/10程度（-2.0）から開始し、調整
