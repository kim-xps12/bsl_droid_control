# V24: contact_no_vel 3D化 + scale増加（接地時かかと叩き対策）

## 概要

V23の目視評価で、足裏が地面に接地した瞬間にankle pitchが小刻みに振動し、かかとで地面を繰り返し叩く動作（heel tapping）が観察された。2つの専門エージェント（データ分析 + ロボティクス/物理学）による多角的分析の結果、以下の2つの複合原因が特定された:

1. **PD制御器のアンダーダンピング**（kp=35, kd=2, ζ≈0.44）: 着地衝撃に対するankle pitch応答が減衰不足で振動を増幅
2. **contact_no_velのZ方向欠落**: XY方向のみをペナルティ対象としており、Z方向のかかと叩き動作を捕捉できていない

V24では、ポリシー側からの対策として、contact_no_velをXYZ 3D化しscaleを3倍に増加する。PD制御器のアンダーダンピング（実機チューニング相当の問題）は、V24の効果が不十分な場合にV25でankle限定kd増加として対処する。

## 前バージョンからの改善事項

### V23の課題

V23は報酬バランスの健全化に成功（ratio 0.070）し、歩行基本構造を完全に維持した。しかし目視評価で以下の動作品質の問題が新たに特定された:

1. **接地時のかかと叩き**: stance phase開始時にankle pitchが~4回/gait cycle振動（速度RMS 3.75-4.00 rad/s）
2. **contact_no_velのXY限定**: Z方向の足速度（かかと叩きの本体）がペナルティ対象外
3. **contact_no_vel scaleの不足**: -0.1はcontact報酬(+0.4)に対して無視できるレベル（寄与率2.2%）

### V23レポートの次バージョン提案との関係

V23レポートではOption A（base_vel_y scale微調整）を推奨していたが、目視評価で新たに特定されたかかと叩き問題への対策を優先した。V23提案のOption A-Cは今後のバージョンで検討する。

### パラメータ変更表

| パラメータ | V23 | V24 | 変更理由 |
|-----------|-----|-----|----------|
| `contact_no_vel` 定義 | XY速度のみ (`:2`) | XYZ 3D速度 (`:3`) | Z方向のかかと叩きを捕捉するため。env側`_reward_contact_no_vel`を修正 |
| `contact_no_vel` scale | -0.1 | -0.3 | 3x増加。contact報酬(+0.4)に対して実効的な抑制力を確保 |

## 設計詳細

### contact_no_vel 3D化の根拠

V23のcontact_no_vel報酬関数:
```python
# 変更前（V23まで）
vel_xy = self.feet_vel[:, :, :2]          # XY方向のみ
vel_sq = torch.sum(torch.square(vel_xy), dim=2)

# 変更後（V24）
vel_xyz = self.feet_vel[:, :, :3]         # XYZ方向全て
vel_sq = torch.sum(torch.square(vel_xyz), dim=2)
```

データ分析で確認されたstance中のankle pitch振動:
- 速度ゼロ交差: 36-39回/全stance期間（~4回/gait cycle）
- 速度RMS: 3.75-4.00 rad/s（正常値は0.5-1.0 rad/s）
- 最大速度: 10.31-10.85 rad/s

これらはZ方向の足速度として計測される。XY方向のみのペナルティではこの振動を抑制できない。

### scale増加の根拠

V23のcontact_no_vel寄与:
- ペナルティ値: -0.0046（全ペナルティの2.2%）
- contact報酬: +0.7005
- **contact_no_vel / contact比 = 0.66%**（ほぼ無視される水準）

scale 3x増加（-0.1→-0.3）+ Z方向追加により:
- 推定ペナルティ値: -0.015〜-0.025（寄与率5-10%程度を見込む）
- ratio影響: 0.070→~0.080（健全域内に留まる見通し）

### 報酬構成（V24）

V23と同一の17項目。変更はcontact_no_velの定義（env側）とscaleのみ。

## 考察と改善案

※以下はデータ分析エージェント（接地タイミング・非対称性・時系列の精密分析）とロボティクス/物理学エージェント（PD制御理論・接触力学・運動連鎖の俯瞰分析）の対話的分析を統合したものである。

### 1. かかと叩きが目視で改善しない根本原因

#### 1.1 contact_no_velが測定しているものの構造的欠陥

**contact_no_vel報酬関数は、かかと叩きの原因であるankle pitch振動を構造的に測定できていない。**

env側コード（`droid_env_unitree.py` L780-）の実装:
```python
vel_xyz = self.feet_vel[:, :, :3]  # foot LINKのデカルト並進速度（リンクフレーム原点）
```

`self.feet_vel`は`robot.get_links_vel()`から取得した**足リンクフレーム原点の並進速度**（ワールド座標系）である。足リンクフレーム原点はankle関節位置にあるため、この速度はankle関節の**親リンク（shank）の運動**に由来する。

一方、かかと叩きの本体はankle pitch関節の**回転振動**であり、これは足リンクの**角速度**として現れる。ankle pitchが角速度ω_ankleで振動するとき、接触点（ankle関節からd≈0.065m離れたかかと）の速度は:
```
v_heel = ω_ankle × d ≈ ω_ankle × 0.065 m
```
だが、この速度はリンクフレーム原点の並進速度には含まれない。つまり **contact_no_velはankle pitch回転による接触点速度を捕捉していない**。

V24のXYZ 3D化で追加されたZ方向成分は、shankのバウンシング（鉛直方向のリンク原点移動）を捕捉するが、ankle pitch回転によるかかと叩きとは無関係である。

#### 1.2 かかと叩きはPRE-CONTACT現象（接地前加速）

データ分析により、かかと叩きの本体は接地前（swing phase末期）にあることが判明:

| タイミング | V23 L |vel| | V24 L |vel| | 変化 |
|-----------|---------|---------|--------|
| t-3（スイング中） | 4.03 rad/s | **6.72 rad/s** | **+66.8%悪化** |
| t-2（スイング中） | 3.47 rad/s | **6.50 rad/s** | **+87.7%悪化** |
| t-1（接地直前） | 2.53 rad/s | **4.79 rad/s** | **+89.8%悪化** |
| t=0（接地瞬間） | 0.99 rad/s | 2.26 rad/s | +128.5% |
| t+1（スタンス） | 0.45 rad/s | 1.07 rad/s | +136.3% |

contact_no_velはスタンス中（t≥0）のみペナルティを適用する。V24のポリシーは:
1. スタンス中の速度を抑制（測定値: stance vel std -24〜-28%）→ **これが"ankle速度改善"の正体**
2. 代償として**スイング末期の速度を大幅に増加**（t-1で+89.8%）
3. 接地前の足位置降下量がV23の8.91°→V24の16.86°に**+89.2%増加**

結果: 足はV23より**高速で**地面に到達し、接地衝撃によるリバウンド（かかと叩き）はそのまま残る。

#### 1.3 接地後リバウンドは構造的に不変

| 指標 | V23 | V24 |
|------|-----|-----|
| 接地後ゼロ交差回数（0.2s窓） | ~2.0回 | ~2.0回 |
| Lリバウンド振幅 | 5.99° | 4.94° |
| Rリバウンド振幅 | 6.18° | **6.98°** (+12.9%) |
| Rリバウンドタイミング | 1.7 steps | **3.0 steps** |

接地後の振動回数は不変。Rfoot のリバウンドは振幅増加+遅延化しており、目視でより目立つ動きになった。

#### 1.4 かかと叩きの真の原因: PDアンダーダンピング

かかと叩きは**ポリシーの問題ではなく物理/PD制御の問題**である:

- PD制御器: kp=35, kd=2（全関節共通）
- ankle pitch接触時の実効慣性: I_eff ≈ 0.03-0.05 kg·m²（body chain反映）
- 固有振動数: ω_n = √(kp/I_eff) ≈ 26-34 rad/s ≈ **4.2-5.4 Hz**（観測値~5Hzと一致）
- 減衰比: ζ = kd/(2√(kp·I_eff)) ≈ **0.76-0.90**（ζ<1でアンダーダンプ）

接地衝撃（インパルス外力）に対してζ<1.0のPD系は振動的応答を示す。ポリシー制御周波数50Hzに対してankle振動~5Hzは10ステップ/サイクルだが、action latency（20ms）を含めると実効応答は40ms遅延=振動の0.2サイクル分。**ポリシーは振動を観測してから対応するまでに振動が既に進行しており、能動的ダンピングは不可能**。

さらにGenesisのインピーダンスベース接触モデルにより、接触バネとPDの結合振動が発生し、実効的な減衰はPDパラメータ単体の予測より低くなる。

**結論: 報酬整形ではかかと叩きは解消できない。物理パラメータ（ankle kd増加）による直接的対策が必要。**

### 3. 報酬設計の体系的レビュー

- ratio 0.073は健全域（0.070-0.085）に留まり、報酬バランス自体は崩れていない
- 問題は報酬バランスではなく、contact_no_velが**測定対象を捕捉できていない**構造的欠陥
- 正報酬が全カテゴリで+5-10%増加しているが、これはstance中のリンク並進速度低減の反映であり、実際の歩行品質改善を意味しない
- **報酬整形によるかかと叩き対策はCLOSED**: contact_no_velはリンク並進速度を測定しており、ankle回転速度を原理的に捕捉できない。scale増加は副作用のみ

### 4. 改善提案

#### Option A（推奨）: ankle_pitch限定kd増加 + contact_no_vel V23復帰

- 変更1: ankle_pitchのkd: 2.0 → **5.0**（env側で関節別kdをサポート）
- 変更2: contact_no_velを**V23設定に復帰**（XYのみ、scale -0.1）
- 根拠:
  - kd=5.0でζ≈1.0-1.3（概臨界〜過減衰）。接地衝撃後の振動を1-2サイクル以内に収束させる
  - 報酬項目数は17のまま（物理パラメータ変更のため）
  - contact_no_vel復帰により非対称化カスケードを解消
  - 実機のRobStride RS-02は逆起電力ダンピングにより実質的にkd>2相当。**kd増加はsim-to-realギャップ縮小にも寄与**
- 期待効果: かかと叩き抑制（根本原因対策）、Yawドリフト回復、hip_pitch相関回復
- リスク: kd増加によりankle応答が鈍化し、歩行適応性が低下する可能性。ただしankle_pitchのaction_scale 0.25 radはkd=5でも十分な制御権限を確保
- 実装: `droid_env_unitree.py`の`set_dofs_kv`を関節別対応に修正（小規模コード変更）

#### Option B: ankle_pitch限定kd増加 + contact_no_vel 3D維持（scale -0.1）

- 変更1: ankle_pitch kd: 2.0 → 5.0
- 変更2: contact_no_vel 3D維持、scale -0.3→-0.1
- 根拠: 3D定義自体は足滑り防止として合理的（XY方向の効果は有効）。scaleを-0.1に戻すことでカスケード回避
- リスク: 2パラメータ同時変更（kd + scale）で効果分離困難。1変更1検証原則に違反
- **Option Aより劣る**: kd変更がかかと叩きの直接対策。contact_no_vel変更を同時に行う必要性がない

#### Option C: URDF ankle damping増加

- 変更: URDF ankle_pitch jointの`<dynamics damping="0.1">`を`damping="3.0"`に増加
- 根拠: URDF dampingは物理サブステップレベル（100Hz）で適用され、kd（50Hz制御ステップ）より高頻度
- リスク: URDF変更は全実験に影響。env側kd変更のほうが訓練スクリプト単位で管理可能

### 5. 推奨方向性

**Option Aを強く推奨**。理由:

1. **根本原因への直接対策**: かかと叩きはPDアンダーダンピングが原因。kd増加は唯一の直接対策
2. **報酬整形のPareto限界を超える**: contact_no_velは測定対象の構造的欠陥により、いかなるscale調整でもかかと叩きを解消できない。これはV4-V12で内股のPareto限界に達したのと同種の問題
3. **副作用最小**: 物理パラメータ変更は報酬項目数・バランスを変えない。非対称化カスケードのリスクがない
4. **1変更1検証原則**: kd変更のみの効果を検証。contact_no_velはV23に完全復帰
5. **sim-to-real整合性**: 実機のモータダンピングに近づけることでシミュレーション忠実度が向上

## まとめ

### 達成されたこと

- contact_no_vel寄与率が2.2%→5.9%に上昇し、リンク並進速度ペナルティとしては機能
- 歩行周波数1.25Hz、内股±9.4°、基本歩行構造は維持
- 正報酬が全カテゴリで+5-10%増加

### 達成されなかったこと

- **かかと叩きは目視で未改善**（ankle vel std改善は接地前加速の代償でstance中のみ）
- Yawドリフトが+11.18°（V23: -4.46°）に大幅悪化
- hip_pitch相関が-0.719（V23: -0.846）に-15.0%悪化
- Pitch stdが1.37°（V23: 0.81°）に+69.1%悪化
- 横方向R²が0.448（V23: 0.735）に-39.1%崩壊
- knee_pitch非対称度が-11.7%（V23: +0.9%）に悪化

### 教訓

- **contact_no_velの構造的欠陥**: `feet_vel`はリンクフレーム原点の並進速度であり、ankle pitch回転による接触点速度を含まない。かかと叩きはankle回転振動が原因であるため、contact_no_velではいかなるscale/次元拡張でも原理的に捕捉不可能
- **かかと叩きは報酬整形の限界を超える問題**: PDアンダーダンピング（ζ<1.0）による物理現象。ポリシーの制御帯域（50Hz, +20ms latency）では~5Hz振動を能動的にダンプ不可能
- **報酬がstance中のみ適用される罠**: ポリシーはswing末期に加速→stance開始後に減速する回避戦略を採用。「報酬値の改善」≠「目視品質の改善」
- **MLPの非対称補償パターン（V18再発）**: 強い制約下でMLPが対称解のサドルポイントから脱出し、非対称局所最適に到達。ankle L/R非対称倍増→Yawドリフト
- **Physical structure > reward shaping（サーベイ8.1原則の再確認）**: V1（narrow URDF）、V20（gait_frequency）に続き、物理パラメータ変更が報酬整形を上回る事例

## 備考

- V23レポートの提案（Option A: base_vel_y調整）ではなく、目視評価で新たに特定されたかかと叩き問題への対策を優先
- contact_no_velの定義変更（XY→XYZ）はenv側の1関数修正のみ。他の報酬関数への影響なし
- PD制御器のアンダーダンピング問題（ζ≈0.44）はV24では未対処。効果不十分の場合、V25でankle限定kd増加（2.0→5.0）を検討
