# EXP009: 報酬設計原則

exp009ではexp008で確立された報酬設計を継承する。Unitree RL Gymの報酬設計に基づく理論的背景は [exp007_unitree_rl_gym_survey.md](../exp007_droid_rl_walking_ref_unitree/exp007_unitree_rl_gym_survey.md) を参照。

exp007の40バージョンにわたる知見の総括は [exp007_summary_for_exp008.md](../exp007_droid_rl_walking_ref_unitree/exp007_summary_for_exp008.md) を参照。

## 1. 報酬項目追加に関する厳格ルール（exp007 V10教訓に基づく）

**exp007 V1-V10の試行錯誤から得られた重要な教訓**を今後の実験で必ず遵守すること。

### 報酬項目の追加は最終手段

```
❌ 問題発生 → 新しい報酬項目を追加
✅ 問題発生 → 既存項目の重み調整 → 効果なし → 項目の削除を検討 → 最後に追加を検討
```

**根拠**：
- exp007 V3-V4（16-17項目）で0.15-0.19 m/sを達成
- exp007 V7-V10（22項目）で不安定化、最終的に静止ポリシー（0.006 m/s）
- Unitree G1/H1は13-14項目で実機歩行を達成
- **項目を増やすほど良くなるわけではない。むしろ項目間の相互作用が予測困難になる。**

### 新規項目追加前の必須チェックリスト

新しい報酬項目を追加する前に、以下を**全て**確認すること：

- [ ] **既存項目の重み調整で対応できないか？**
- [ ] **不要な項目を削除して簡素化できないか？**
- [ ] **Unitree RL Gymに同等の項目が存在するか？**（存在しない場合は慎重に）
- [ ] **exp007 V3-V4（動作成功時）に存在した項目か？**
- [ ] **追加による副作用（ペナルティ累積、相互干渉）を十分に検討したか？**
- [ ] **一度に追加するのは1項目のみか？**

### 避けるべきパターン（exp007で実証済み）

| パターン | 問題 | 実例 |
|---------|------|------|
| 複数項目の同時追加/強化 | 相互作用が予測困難 | V10: 5項目同時変更 → 静止ポリシー |
| 対称性報酬（symmetry）の強化 | hip_pitch同期誘発 | V9: symmetry 1.0 → 相関+0.605に悪化 |
| ペナルティの累積強化 | 動くこと自体がペナルティに | V7→V10でペナルティ倍増 → 静止 |
| 複雑な交互歩行報酬 | 意図しない効果 | alternating_gait/hip_pitch_antiphase → 効果なし |
| hip_pos compound penalty | hip_yaw+hip_roll同時制約 | V37-V40: 独立制御不可 |
| symmetry_range | 振幅縮小(race to bottom) | V33: 等化ではなく全体縮小 |

### 推奨するアプローチ

1. **シンプルさを維持**: 15-17項目を上限とする
2. **実績ベース**: exp007 V39、Unitreeの設定を参照
3. **1変更1検証**: 一度に変更するのは1-2項目まで
4. **削除優先**: 追加より削除を優先的に検討
5. **ペナルティ控えめ**: 主報酬を上回らないよう設定

## 2. 報酬項目数の目安

| 項目数 | 評価 | 根拠 |
|-------|------|------|
| 13-14 | ✅ Unitree推奨 | G1/H1で実機歩行達成 |
| 15-17 | ✅ BSL-Droid推奨 | exp007 V3-V4で0.15-0.19 m/s達成 |
| 18-20 | ⚠️ 注意 | 相互作用リスク増加 |
| 21以上 | ❌ 非推奨 | exp007 V7-V10で不安定化実証済み |

## 3. 階層的報酬設計の原則

報酬スケールは以下の階層構造を遵守すること：

1. **主報酬**（最優先）：tracking_lin_vel（1.0-1.5）> tracking_ang_vel（0.5）
2. **歩行品質報酬**（次優先）：feet_air_time, contact, single_foot_contact
3. **補助報酬**（低優先）：alive, その他
4. **ペナルティ**（抑制のみ）：**絶対値の合計が主報酬を上回らないこと**

**重要**: ペナルティの累積が主報酬を上回ると、「動かない」が最適解になる（exp007 V10で実証）。

## 4. exp009固有の考慮事項

### 4.1 全方向コマンドに伴う報酬設計変更

exp009ではexp008の前進のみ（X: [0.25, 0.35]）から全方向（X/Y: [-0.3, 0.3]）に拡張。これに伴い:

- **base_vel_y無効化**: Y方向コマンドと矛盾するため。tracking_lin_velがY方向追従を含む
- **velocity_deficit多方向対応**: コマンド方向への射影ベースに汎化。後退コマンドでも正しく機能
- **step_length多方向対応**: コマンド方向への足間距離に汎化。横歩きでも意味のある歩幅を計測
- **foot_lateral_velocity**: 既にコマンド方向ベクトルを使用。変更不要

## 5. トラブルシューティング

### 5.1 「動かない」ポリシーが学習された（静止ポリシー問題）

exp008 `exp008_reward_design.md` Section 5.1 を参照。

### 5.2 方向別の追従精度にバラつきがある

exp009固有の新課題。原因候補:

1. **報酬関数の方向非対称性**: 一部の報酬が特定方向に偏っている可能性
2. **初期姿勢の影響**: 前傾姿勢が前進に有利・後退に不利
3. **PDクランプの影響**: hip_roll inward limitが横歩きに影響する可能性

対処:
1. 方向別の追従精度を測定し、弱い方向を特定
2. 報酬関数の方向依存性を確認
3. 必要に応じてコマンド範囲を方向別に調整
