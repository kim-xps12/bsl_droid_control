# V4: is_moving判定のバグ修正

## 概要

V3で観察されたLFT追従率41.5%、RGT追従率27.0%の根本原因として、`droid_env_unitree.py`の報酬関数における`is_moving`判定のバグを特定・修正する。`single_foot_contact`等の報酬関数が`commands[:, 0].abs() > 0.05`（X方向のみ）を使用していたため、純粋な横方向コマンドで歩行品質報酬が完全無効化されていた。

**目視評価結果**: 静止状態を維持できず、歩行時に脚がジタバタと振動しながら動く危険な挙動が観察された。

## 前バージョンからの改善事項

V3レポートの「次バージョンへの提案」ではmax_iterations 4000を推奨したが、方向別性能の非対称性がイテレーション不足ではなくコードバグに起因することを特定したため、バグ修正を優先する。

| パラメータ | V3値 | V4値 | 変更理由 |
|-----------|------|------|---------|
| `_reward_single_foot_contact` is_moving | `commands[:, 0].abs() > 0.05` | `torch.norm(commands[:, :2], dim=1) > 0.1` | 横歩き時に報酬が無効化されるバグ |
| `_reward_hip_pitch_antiphase` is_moving | 同上 | 同上 | 将来向け一括修正（現在scale=0で無効） |
| `_reward_hip_pitch_antiphase_v2` is_moving | 同上 | 同上 | 将来向け一括修正（現在scale=0で無効） |
| `_reward_both_legs_active` is_moving | 同上 | 同上 | 将来向け一括修正（現在scale=0で無効） |
| max_iterations | 2000 | 2000 | 変更なし（バグ修正効果の分離） |

## 設計詳細

### バグの詳細

`droid_env_unitree.py`の4つの報酬関数で「移動コマンドかどうか」の判定が以下のようになっていた:

```python
# バグ: X方向のみを参照
is_moving = self.commands[:, 0].abs() > 0.05
```

純粋な横方向コマンド `[0.0, 0.3, 0.0]`（Left歩行）の場合:
- `commands[:, 0]` = 0.0（X方向速度）
- `0.0 > 0.05` = **False**
- → `single_foot_contact`報酬が**完全無効化**

修正後:
```python
# 修正: XYノルムで判定（多方向対応）
is_moving = torch.norm(self.commands[:, :2], dim=1) > 0.1
```

閾値0.1は既存の正しい実装（`feet_air_time`, `swing_duration`, `swing_contact_penalty`）と統一。

### 影響範囲

- **V4で直接影響**: `single_foot_contact`（scale=0.5、ACTIVE）のみ
- **将来向け修正**: `hip_pitch_antiphase`, `hip_pitch_antiphase_v2`, `both_legs_active`（全てscale=0で無効）
- **後方互換性**: exp007/exp008は前進のみ（X > 0.25 m/s）のため、修正による影響なし

### 報酬構成

V3と同一の16項目。変更なし。

## 考察と改善案

### 目視評価所見

目視評価において以下が観察された:
- 静止状態を維持できない（常に脚が動いている）
- 歩行時に脚がジタバタと高周波振動しながら移動
- 全体として危険な印象の歩行

### 対話による多角的分析

以下、データ分析専門エージェント（DA）とロボティクス・物理学俯瞰エージェント（Lead）の対話形式で考察を行う。

---

**DA**: まず、ジタバタ挙動の定量的な特徴を整理します。全関節のFFT分析で最も顕著な変化は、L/R両方のhip_yaw関節の主周波数が3.62 Hzに遷移したことです。V3ではL_hip_yawが0.12 Hz、R_hip_yawが1.25 Hzだったのが、V4では両方とも歩行周波数（1.25 Hz）の約3倍の振動を示しています。高周波成分比率（>2Hz）もL_hip_yaw 62.5%（V3: 43.4%、1.44倍）、R_hip_yaw 72.1%（V3: 43.0%、1.68倍）と増大しています。

**Lead**: hip_yawの3.62 Hz振動は非常に示唆的です。歩行周波数1.25 Hzの約3倍ということは、1歩行サイクル中に3回の振動が入っていることになります。これは自然な歩行パターンではなく、ポリシーがhip_yawを「微小な方向修正」に使用している可能性があります。exp008のV18で観察されたのと同じパターンですね——過度な制約が加わると、ポリシーはhip_yaw compensationに逃げるという因果連鎖です。

**DA**: その仮説と整合する重要なデータがあります。アクション変化率分析で、action_1（L_hip_roll）だけが+32%と突出して悪化しています。他の9アクションの変化率はV3比で±6%以内です。また、DOF速度のホットスポットとしてR_hip_roll速度RMSが1.52倍、ピーク角速度が2.97倍になっています。つまりジタバタは**全身的ではなく、hip_yaw（両側）とhip_roll（非対称にL actionとR velocity）に局在**しています。

**Lead**: 局在性は重要な手がかりです。is_moving修正の直接影響は`single_foot_contact`報酬（scale=0.5）のみです。この報酬は「片足のみが接地している」状態を報酬化します。V3では横方向コマンド時にこの報酬が無効化されていたため、ポリシーは横歩き時に「足を交互に上げる」インセンティブを持っていませんでした。V4でこれが有効化されたことで、**横歩き時にも片足接地を最大化しようとする新たな圧力**が加わりました。

**DA**: しかし奇妙なことに、接触切替頻度はV4がV3より**低い**のです。遷移レートは8.27/s対9.27/s（V4/V3 = 0.89）。マクロな接触パターン（片足接地94.2%）もV3と同一です。ジタバタは接触切替ではなく、**接触状態を維持したまま関節が振動している**ことを示唆します。

**Lead**: なるほど。つまり足は地面に着いているが、脚の関節（特にhip_yawとhip_roll）が振動している。これは接触パターン問題ではなく、**姿勢制御ストラテジー**の問題です。考えられるメカニズムは以下の通りです:

1. is_movingバグ修正により、横方向コマンド時のsingle_foot_contact報酬が有効化
2. ランダムコマンドの約50%はX成分が小さく（|vx| < 0.15）、V3ではこれらでsingle_foot_contactが部分的に無効化されていた
3. V4ではXYノルムで判定するため、ほぼ全てのコマンドでsingle_foot_contactが有効
4. 結果として、ポリシーは**常に片足接地を意識する**必要があり、バランス維持のためにhip_yawとhip_rollを高頻度で微調整

**DA**: この仮説を検証するデータがあります。報酬コンポーネント比較で、single_foot_contactは0.4861（V3: 0.4674、+4.0%）と微増に留まっています。一方、ペナルティ側では**action_rate +24.6%**、**ang_vel_xy +17.6%**、**lin_vel_z +19.9%**が目立ちます。特にaction_rateの悪化は、高頻度関節振動の直接的な反映です。

**Lead**: action_rate penaltyが-0.005→-0.039（V4実測値）で、single_foot_contactの実測値0.486と比べると**12.5倍の差**があります。action_rate scale -0.005は元々exp008で前進歩行用に設定されたもので、全方向歩行で新たに生じた高頻度振動を抑制するには桁違いに小さいのです。

## まとめ

exp009 V4はV3の`is_moving`バグを修正し、横方向コマンド時にも`single_foot_contact`報酬を正しく有効化した。訓練報酬は58.93（V3: 59.89、-1.6%）、Last 50 stdは0.376（V3: 0.228、+64.9%）と収束不十分。

方向別評価では、LFTのYawドリフトが+103.2°→+5.5°と劇的に改善し、BWDの転倒も解消された（3/3→0/3）。一方、全4方向の追従率が低下（平均52.0%、V3: 56.2%）し、RGTは19.7%まで悪化した。

最大の問題は目視で確認された**ジタバタ挙動**である。定量分析の結果、L/R hip_yawが3.62 Hz振動（V3の0.12-1.25 Hzから遷移）、L_hip_rollアクション変化率+32%、R_hip_roll速度RMS 1.52倍が主因と特定された。これらは全身的ではなく**hip_yawとhip_rollに局在**した振動であり、is_moving修正によるsingle_foot_contact有効化範囲の拡大が、ポリシーの姿勢制御ストラテジーを変化させた結果と考えられる。

バグ修正自体は正しいが、報酬ランドスケープの実質的変更に2000 iterationsでは適応不足の可能性がある。V5ではmax_iterationsを4000に増加し、現在の報酬構造での十分な収束を目指す。

## 備考

- 本バージョンの変更は`droid_env_unitree.py`の環境コード修正のみ。訓練パラメータ（報酬scale、コマンド範囲、max_iterations）はV3と完全同一
- V3レポートの推奨（max_iterations 4000）はバグの存在を認識していない状態での判断であり、バグ修正が優先される
- 1変更1検証原則に基づき、max_iterationsは2000に据え置き。バグ修正効果をV3と直接比較可能にする
- 方向別評価のseed間で決定論的シミュレーションにより同一結果が発生（V3 RGT: 全seed同一、他は一部seed同一）。報告されるstdは実効seed数の制約を受ける
- is_moving修正はコマンド空間の約50%（横方向寄り）でsingle_foot_contact報酬を新たに有効化するため、「1パラメータのバグ修正」ではあるが実効的には報酬ランドスケープの構造的変更に相当する
