# V1: 全方向速度コマンド対応（exp008 V25ベース）

## 概要

exp008 V25をベースに、速度指令を前進のみから前後左右4方向に拡張する。exp009の初回実験として、多方向コマンド追従歩行の基盤を確立することを目指す。

## 前バージョンからの改善事項

exp008 V25（前進歩行のみ）から、全方向速度コマンドへの対応を追加。

| パラメータ | exp008 V25 | exp009 V1 | 変更理由 |
|-----------|-----------|-----------|---------|
| lin_vel_x_range | [0.25, 0.35] | [-0.3, 0.3] | 前後対称コマンドに拡張 |
| lin_vel_y_range | [0, 0] | [-0.3, 0.3] | 左右並進コマンドを追加 |
| ang_vel_range | [0, 0] | [0, 0] | V2以降で追加予定 |
| base_vel_y scale | -1.0 | 0 | Y方向コマンドと矛盾するため無効化 |

## 設計詳細

### 速度コマンドのランダマイズ

各環境インスタンスに対し、4秒ごとに新しい速度コマンドが一様分布からサンプリングされる:
- X方向: U[-0.3, 0.3] m/s（前進・後退）
- Y方向: U[-0.3, 0.3] m/s（左右並進）
- Yaw: 0 rad/s（固定）

### 報酬関数の多方向対応

以下の報酬関数を多方向コマンド対応に汎化（環境ファイル側で修正済み）:

1. **velocity_deficit**: X方向固定 → コマンド方向への射影ベース
2. **step_length**: X方向足間距離 → コマンド方向足間距離
3. **base_vel_y**: 無効化（tracking_lin_velがY方向追従を含む）
4. **foot_lateral_velocity**: 変更不要（既にコマンド方向ベクトルを使用）

### 報酬構成

有効報酬項目数: 16（V25の17からbase_vel_y無効化）

## 実行コマンド

```bash
cd rl_ws
uv run python biped_walking/train/droid_train_omni_v1.py --max_iterations 500
```

## 評価コマンド

```bash
cd rl_ws
uv run python biped_walking/biped_eval.py -e droid-walking-omni-v1 --no-viewer --duration 10
```

### 収束分析

500 stepsでは収束不十分。exp008 V25（前進のみ）との比較:

| 指標 | exp008 V25 | exp009 V1 | 判定 |
|------|-----------|-----------|------|
| Last 50 mean | 61.37 | 53.12 | -13.4% |
| Last 50 std | 0.10 | 1.02 | 10倍（未収束） |
| Last 50 vs Prev 50 改善 | +0.52（飽和） | +3.15（上昇中） | 未飽和 |
| Q3→Q4 増分 | — | +7.00（加速中） | 後半で学習加速 |

収束不足の根拠:
- **Last 50 std = 1.02**: V25の0.10と比べ10倍の変動。報酬がまだ安定していない
- **Last 50 vs Prev 50 = +3.15**: 傾きが有意に正。学習がまだ進行中
- **Q3→Q4で+7.00（Q2→Q3の+2.87から加速）**: 後半で学習が加速しており、500 stepsで打ち切ったことで学習途上のポリシーを評価している可能性が高い

原因: 全方向歩行（前後左右4方向）は前進のみと比べてタスク複雑度が大幅に増加。同じ500 stepsでは獲得すべき歩容パターンをカバーしきれない。750〜1000 stepsが必要と推定される。

ただし、velocity_deficit の構造的問題（+2650%）が存在するため、iteration増加だけでは根本解決にならない可能性がある。V2でvelocity_deficit scaleを修正した上でiterationを増やすのが合理的（下記「次バージョンへの提案」参照）。

## 考察と改善案

### 成功点

- **訓練収束**: 報酬は安定的に上昇し、エピソード長1001（最大長）に到達。ポリシーは倒れずに歩行を維持できている
- **内股制御の維持**: PD target clamp が多方向歩行でも機能。stance hip_roll ±9.9-10.3°で exp008 V25 の ±9.2-9.3° と同等
- **Roll安定性の維持**: Roll std 4.22°（V25: 4.44°）と微改善。基本的な左右バランス制御は維持
- **接地パターン**: 片足接地91.8%で交互歩行の基本構造は維持。両足空中2.0%は低く安全

### 課題

1. **velocity_deficit の爆発的増加 (+2650%)**: 前進のみで最適化された velocity_deficit（scale -0.5）が、多方向コマンドで構造的に発火し続ける。exp008 V25 では -0.0008 とほぼゼロだったペナルティが -0.0220 に急増。これが因果チェーンの起点となり、ポリシーが歩行品質を犠牲にして速度追従を優先する行動を誘発している
2. **hip_pitch相関の崩壊 (-0.851 → -0.254)**: L/R逆位相歩行パターンが大幅に劣化。ポリシーが多方向コマンドへの対応として非対称な歩行パターンを採用している
3. **L/R歩行周波数の非対称化 (L=1.25/R=1.12 Hz)**: exp008 V20 で確立された均一な歩行周波数が崩れている。hip_pitch相関の崩壊と関連
4. **Yawドリフト +39.24° (4.08°/s)**: L/R歩行非対称による角運動量不均衡が原因。exp008 V25の+2.40°から劇的に悪化
5. **action_rate +231.5%**: ポリシーの過剰応答。多方向コマンドへの対応に苦慮していることの間接的指標
6. **Pitch std +136% (0.78° → 1.84°)**: 前後方向の姿勢安定性が低下。後退コマンドへの対応が一因と考えられる

### 次バージョンへの提案

**推奨（V2）: velocity_deficit scale -0.5 → -0.1**

根拠:
- velocity_deficit は因果チェーンの起点であり、+2650% の爆発的増加がペナルティ負荷を押し上げている
- exp008 V25 で velocity_deficit はほぼゼロ（-0.0008）であり、前進歩行品質への寄与は限定的だった。多方向化で突然活性化した項目
- tracking_lin_vel (scale 1.5, Gaussian) が速度追従の主要インセンティブとして十分機能するため、velocity_deficit 縮小のリスクは限定的
- exp008 V23 で実証された「scale tuning > item addition/deletion」原則に基づく段階的対応
- 完全削除（-0.5 → 0）ではなく5倍縮小（-0.5 → -0.1）で効果を検証

期待効果:
- velocity_deficit ペナルティ: -0.0220 → ~-0.0044（80%減）
- ratio の健全域（0.07-0.08）への回復
- ペナルティ圧力低下により、ポリシーが歩行品質を維持しながら多方向コマンドを学習できる可能性

**V3: iteration増加（500 → 750〜1000 steps）**

根拠:
- V1の収束分析でLast 50 std=1.02（V25の10倍）、Q3→Q4で+7.00の加速を確認。500 stepsでは学習途上
- V2でvelocity_deficit scaleを修正してペナルティ構造を安定化した上で、iteration増加の効果を検証
- velocity_deficit修正とiteration増加を同時に行うと効果分離不可（1変更1検証原則）。V2でscale修正→V3でiteration増加の順序が適切

**V4以降の候補**:
- コマンド範囲段階的拡大（カリキュラム学習: Survey 7.3.1）
- ang_vel_range の非ゼロ化（Yaw修正能力の獲得: Survey 7.4.1）
- 対称性報酬の再有効化（L/R非対称化対策）

## まとめ

exp009 V1 は exp008 V25 をベースに全方向速度コマンド対応を追加した初回実験。内股制御（stance ±9.9-10.3°）とRoll安定性（std 4.22°）は維持されたが、2つの主要課題が判明した。第一に、velocity_deficit の構造的不適合（+2650%）を起点とする因果チェーンにより、hip_pitch相関の崩壊（-0.254）、L/R歩行周波数の非対称化、Yawドリフト（+39.24°）が発生。第二に、500 stepsでは収束不十分（Last 50 std=1.02、Q3→Q4で+7.00加速中）であり、全方向歩行のタスク複雑度に対して学習量が不足している。V2 では velocity_deficit scale を -0.5 → -0.1 に低減してペナルティ構造を安定化し、V3 で iteration を 750〜1000 steps に増加して収束を確認する。

## 備考

- 本バージョンはexp009の初回実験。exp008 V25の設定を基盤とし、速度コマンドの多方向対応のみを変更
- Yaw旋回指令はV2以降で追加予定（1変更1検証原則に基づく段階的導入）
- 評価時のコマンドはランダムサンプリングのため、各評価実行で指標値が変動する。特にX速度・Yawドリフトは発行コマンドに依存
- biped_eval.py に `droid-walking-omni-v*` パターンの DroidEnvUnitree ルーティングを追加（data-collector による修正）
