# EXP007: ワークフロー（新バージョン作成・訓練後分析）

本ドキュメントは、新バージョンの作成から訓練後分析までの End-to-End ワークフローを定義する。

```
Phase 1: 準備（Section 2）
    - 前バージョンのレポート参照・精査
    - トレーニングスクリプト作成（コピー + 最小変更）
    - lint・型チェック
    - レポートスケルトン作成
    ↓
Phase 2: 訓練
    - uv run python biped_walking/train/droid_train_unitree_v{N}.py --max_iterations 500
    - 訓練完了を待機
    ↓
Phase 3: 訓練後分析（Section 3）
    - Step A: データ収集（評価・ログ分析・CSV分析・報酬コンポーネント比較）
    - Step B: 定量分析（比較表作成・報酬バランス評価）
    - Step C: サーベイ知見との照合
    - Step D: 独立考察（因果分析・改善案設計）【最重要】
    - Step E: レポート執筆（結果=事実、考察=分析）
    ↓
Phase 4: 完了報告
    - 結果サマリをユーザに提示
    - 次バージョンの推奨案を提示
```

ユーザが「V{N}を行います」「実装を開始しなさい」等の明示的な実行指示を出した場合、Coding Agent はこのフロー全体を自動実行する。

---

## 1. バージョン別レポートのテンプレート

各バージョンのレポート（`exp007_report_v{N}.md`）は以下のテンプレートに従って記述する。

```markdown
# V{N}: {タイトル}

## 概要

{このバージョンで行う実験の目的と概要を記述する。}
{何を達成しようとしているか、どのようなアプローチを取るかを簡潔に説明する。}

## 前バージョンからの改善事項

{前バージョンの課題と、それに対する改善策を記述する。}
{**重要**: 前バージョンのレポート（`exp007_report_v{N-1}.md`）の「次バージョンへの提案」セクションの内容を参照・転記し、それに基づいて改善を行ったことを明記する。}

| パラメータ | 前バージョン値 | 本バージョン値 | 変更理由 |
|-----------|---------------|---------------|---------|
| ... | ... | ... | ... |

## 設計詳細

{報酬設計、パラメータ設定、新規機能などの詳細な設計を記述する。}
{数式や定量的な解説を含めることが望ましい。}

## 実行コマンド

```bash
cd rl_ws
uv run python biped_walking/train/droid_train_unitree_v{N}.py --max_iterations 500
```

## 評価コマンド

```bash
cd rl_ws
uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{N} --no-viewer --duration 10
```

## 結果

{評価を実行した結果を事実のみ記述する。主観的な解釈は含めない。}

### 訓練結果

| 指標 | 値 |
|------|-----|
| 最終報酬 | ... |
| エピソード長 | ... |
| 収束ステップ | ... |

### 評価結果

| 指標 | 値 |
|------|-----|
| X速度 | ... m/s |
| Y速度 | ... m/s |
| hip_pitch相関 | ... |
| ... | ... |

## 考察と改善案

{実験結果に基づく論理的・学術的な考察を記述する。}
{なぜそのような結果になったか、仮説と検証結果を記述する。}
{次バージョンで実施すべき改善案を具体的に記述する。}

### 成功点
- ...

### 課題
- ...

### 次バージョンへの提案
- ...

## まとめ

{レポート全体の要約を記述する。}
{このバージョンで達成したこと、達成できなかったことを簡潔にまとめる。}

## 備考

{もしあれば、補足情報や参考リンクを記述する。}
```

**テンプレート使用上の注意**:
- 全セクションを必ず記載する（該当なしの場合は「該当なし」と記載）
- 結果セクションには主観的解釈を含めず、事実のみを記載する
- 考察セクションでは結果の解釈と次バージョンへの提案を明確に分離する
- 数値データは可能な限り表形式で整理する
- **期待・目標の記述は断定表現を避ける**: 訓練前の期待や目標を記述する際、結果を先取りした断定表現（「〜を達成する」「〜に転換する」）は避け、意図を示す表現（「〜を目指す」「〜を狙う」「〜することを期待」）を使用する

## 2. 新バージョン作成時の手順（Coding Agent向け）【厳守】

新バージョン（V(n+1)）の実装を依頼された際、Coding Agentは以下のチェックリストを**この順番で**実行すること。

### チェックリスト

- [ ] **1. 前バージョンのレポートを参照して精査する**
  - [ ] `exp007_report_v{N-1}.md` を読み込む
  - [ ] 「次バージョンへの提案」セクションの内容を確認する
  - [ ] 課題と改善策を把握する

- [ ] **2. 必要に応じて追加評価を実行する**（任意）
  - [ ] 前バージョンのレポートに不足情報がある場合のみ
  - [ ] 評価: `uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{N-1} --no-viewer --duration 10`
  - [ ] ログ分析: `uv run python scripts/analyze_training_log.py droid-walking-unitree-v{N-1}`

- [ ] **3. 改善策に基づいてトレーニングスクリプトを作成する**
  - [ ] [exp007_rules.md](exp007_rules.md) Section 1.3 トレーニングスクリプトのコピー原則に従う
  - [ ] 必要に応じて環境クラスに新しい報酬関数を追加する（[exp007_rules.md](exp007_rules.md) Section 1.1, 1.2 に従う）
  - [ ] lintチェック・型チェック（コード品質ルールは `AGENTS.md` 参照）

- [ ] **4. 作業中バージョンのレポートを作成する**
  - [ ] `exp007_report_v{N}.md` を新規作成する
  - [ ] 本ドキュメント Section 1 のテンプレートに従って記述する
  - [ ] 「結果」セクションは訓練完了後に記入する旨を明記する
  - [ ] 実装したトレーニングスクリプトに関する事項（設計詳細、パラメータ変更）を詳細に記載する

- [ ] **5. 主レポートにバージョン参照を追加する**
  - [ ] `exp007_droid_rl_walking_ref_unitree.md` のバージョン一覧表に新バージョンを追加する
  - [ ] バージョン進化の概要図も更新する

- [ ] **6. 学習の実行**
  - [ ] ユーザが明示的に実行を指示した場合（例:「V{N}を行います」「実装を開始しなさい」）: 学習を自動実行してよい
  - [ ] 指示が曖昧な場合: 学習コマンドを提示し、ユーザの確認を待つ

### 禁止事項

- ❌ 前バージョンのレポートを確認せずにスクリプトを作成する
- ❌ レポートを作成せずにスクリプトのみを提供する
- ❌ 主レポートへの参照追加を忘れる
- ❌ ユーザの明示的な実行指示がないのにCoding Agentが自動で学習を開始する

## 3. 訓練後の分析・考察フロー（Coding Agent向け）【厳守】

訓練結果のレポートを作成（または改訂）する際、Coding Agentは以下のフローを**この順番で**実行すること。

### フロー

```
Step A: データ収集（評価・ログ分析スクリプトの実行）
    ↓
Step B: 定量分析（収集データの整理・比較表作成）
    ↓
Step C: サーベイ知見との照合（理論的根拠の確認）
    ↓
Step D: 動作悪化の原因分析と改善案の考察（独立考察ステップ）【重要】
    ↓
Step E: レポート執筆（Step A-Dの成果を統合）
```

### 各ステップの詳細

**Step A: データ収集**（[exp007_commands.md](exp007_commands.md) の各スクリプトを使用）
- 評価（基本）: `uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{N} --no-viewer --duration 10`
- 評価（CSV）: `uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{N} --no-viewer --duration 10 --csv`
- CSV時系列分析: `uv run python scripts/analyze_eval_csv.py {N} {N-1}`（[exp007_commands.md](exp007_commands.md) Section 5.4参照）
- 訓練ログ分析: `uv run python scripts/analyze_training_log.py droid-walking-unitree-v{N}`
- 報酬コンポーネント分析: `uv run python scripts/show_reward_components.py droid-walking-unitree-v{N-1} droid-walking-unitree-v{N}`
- 前バージョンとの比較のため、v{N-1}のCSV評価も実行する

**Step B: 定量分析**
- V(N-1)→V(N)の全指標比較表を作成
- 報酬バランスの定量評価（正の報酬合計 vs ペナルティ絶対値合計の比率）
- 報酬項目数のチェック（[exp007_reward_design.md](exp007_reward_design.md) Section 2 の推奨範囲内か）

**Step C: サーベイ知見との照合**
- `exp007_unitree_rl_gym_survey.md`から関連する知見を特定
- 実験結果がサーベイの理論的予測と一致するかを検証

**Step D: 動作悪化の原因分析と改善案の考察**【最重要ステップ】
- Step B（定量分析）とStep C（サーベイ照合）の結果を入力とし、レポート執筆に先立って独立した考察を行う
- **このステップを省略してデータ収集から直接レポート執筆に進むことは禁止**
- 考察の深さはレポートの品質を決定する最重要ステップであり、他のステップと同等以上の工数を割くこと

Step Dで実施すべき考察項目:
1. **動作悪化メカニズムの因果関係の特定**: なぜ動作が悪化したか、報酬コンポーネント間の相互作用の分析
2. **報酬設計全体の体系的整理**: 寄与度ランキング、報酬バランス、削減候補の選定
3. **複数改善案の設計と評価**: 各案の期待効果・リスク・理論的根拠を明記
4. **推奨案の決定**: 1変更1検証の原則を遵守した段階的実施計画

**Step E: レポート執筆**
- Step A-Dの成果を本ドキュメント Section 1 のテンプレートに統合
- 結果セクションには事実のみを記載（Step A-Bのデータ）
- 考察セクションにはStep C-Dの成果を反映

### 禁止事項

- ❌ Step Dを省略してデータ収集から直接レポート執筆に進む
- ❌ 前バージョンのレポートの考察をそのまま流用する（ゼロベースで再考察すること）
- ❌ サーベイ知見を参照せずに改善案を設計する

---

## 4. 中断・再開

- 各 Phase は独立しており、途中で中断・再開が可能
- 訓練が失敗した場合は Phase 1（Section 2）に戻りスクリプトを修正
- ユーザが追加の目視評価を行う場合、Phase 3 の Step A にユーザー目視観察を追加
