### 1.3 報酬関数の詳細解説

V1で使用する15種類の報酬関数について、その数学的定義、物理的意図、定量的解釈を以下に詳述する。

---

#### 1.3.1 速度追従報酬（Task Rewards）

##### tracking_lin_vel（線速度追従）

**数学的定義:**
```python
lin_vel_error = Σ(commands_xy - base_lin_vel_xy)²
reward = exp(-lin_vel_error / σ)  # σ = 0.25
```

**物理的意図:**
- ロボットが指令された水平面内（X, Y方向）の速度を追従することを促す
- 歩行学習における最も基本的なタスク報酬

**定量的解釈:**
- σ=0.25のとき、速度誤差0.25m/sでreward≈0.37（1/e）
- 誤差0m/sでreward=1.0（最大）、誤差1.0m/sでreward≈0.02
- exp関数により、小さな誤差では高報酬、大きな誤差では急激に報酬が減少
- Unitreeはこのスケール1.0を全報酬の基準として設計

**設計根拠:**
- 二乗誤差＋exp関数の組み合わせは、目標近傍での微調整を促進
- 線形ペナルティと比較して、目標から遠いときのペナルティが緩やか

---

##### tracking_ang_vel（角速度追従）

**数学的定義:**
```python
ang_vel_error = (commands_yaw - base_ang_vel_z)²
reward = exp(-ang_vel_error / σ)  # σ = 0.25
```

**物理的意図:**
- ロボットが指令されたYaw軸（鉛直軸）周りの旋回速度を追従することを促す
- 直進歩行時はcommands_yaw=0なので、直進維持の報酬としても機能

**定量的解釈:**
- スケール0.5はtracking_lin_velの半分の重要度
- 角速度誤差0.5rad/sでreward≈0.37
- 旋回コマンドなし（0）の場合、Yawドリフトへのペナルティとして機能

**BSL-Droid適用:**
- exp006で問題となったYawドリフト（29°）の抑制に寄与

---

##### alive（生存報酬）

**数学的定義:**
```python
reward = 1.0  # 毎ステップ定数
```

**物理的意図:**
- エピソードが継続している限り一定の報酬を与える
- 早期終了（転倒等）よりも長く立っていることを優先させる

**定量的解釈:**
- スケール0.1で、dt=0.02秒あたり0.002の報酬
- 20秒エピソードで累積200（1000ステップ×0.1×0.002）
- 他の報酬が負になりがちな学習初期に、ポリシーの安定化に貢献

**Unitreeとの比較:**
- Unitree: 0.15、V1: 0.1（やや控えめ）
- 過度な生存報酬は「動かない」ポリシーを学習するリスク

---

#### 1.3.2 安定性ペナルティ（Stability Penalties）

##### lin_vel_z（Z軸速度ペナルティ）

**数学的定義:**
```python
penalty = base_lin_vel_z²
```

**物理的意図:**
- 胴体の上下動（跳躍や急激な沈み込み）を抑制
- 滑らかな歩行軌道を促進

**定量的解釈:**
- スケール-2.0で、Z速度0.5m/sの場合、ペナルティ=0.5×dt
- 歩行中の典型的なZ速度変動（0.1-0.2m/s）では小さなペナルティ
- 跳躍（>0.5m/s）では強いペナルティ

**設計根拠:**
- 二脚歩行では多少のZ速度変動は自然だが、過剰な上下動はエネルギー効率を低下
- 特にexp006で見られた高周波振動の抑制に効果的

---

##### ang_vel_xy（XY角速度ペナルティ）

**数学的定義:**
```python
penalty = base_ang_vel_x² + base_ang_vel_y²
```

**物理的意図:**
- Roll軸（X）とPitch軸（Y）周りの回転速度を抑制
- 胴体の傾きの急激な変化を防止

**定量的解釈:**
- スケール-0.05は比較的小さい（微調整用）
- 角速度1.0rad/s（約57°/s）でペナルティ=0.05×dt
- 転倒寸前の急激な回転（>2rad/s）では顕著なペナルティ

**設計根拠:**
- 小さいスケールは、歩行に必要な自然な姿勢変動を許容
- 過度に抑制すると硬直した不自然な歩行になる

---

##### orientation（姿勢維持ペナルティ）

**数学的定義:**
```python
# projected_gravity: 胴体座標系での重力ベクトル（正規化済み）
# 直立時は [0, 0, -1]、傾くとXY成分が増加
penalty = projected_gravity_x² + projected_gravity_y²
```

**物理的意図:**
- 胴体を鉛直に保つことを促す
- Roll/Pitchの傾きをペナルティ化

**定量的解釈:**
- 直立時: penalty=0
- 30°傾斜時: sin(30°)²≈0.25 → ペナルティ=0.25×0.5×dt
- 45°傾斜時: sin(45°)²=0.5 → ペナルティ=0.5×0.5×dt

**BSL-Droid調整:**
- Unitree: -1.0 → V1: -0.5（半減）
- 理由: 小型ロボットは慣性が小さく姿勢変動が大きい
- exp006のPitch 13.7°問題に対応

---

##### base_height（胴体高さペナルティ）

**数学的定義:**
```python
penalty = (base_pos_z - target_height)²  # target = 0.20m
```

**物理的意図:**
- 胴体を目標高さに維持
- 沈み込み（膝折れ）や過度な伸び上がりを抑制

**定量的解釈:**
- 目標0.20mから±0.05m逸脱でペナルティ=0.0025×5.0×dt
- ±0.10m逸脱でペナルティ=0.01×5.0×dt
- 二乗関数により、小さな逸脱は許容、大きな逸脱は強くペナルティ

**BSL-Droid調整:**
- Unitree: target=0.78m, scale=-10.0
- V1: target=0.20m, scale=-5.0
- 脚長の比率（0.23m/1.0m≈0.23）から目標高さを設定

---

#### 1.3.3 歩行品質報酬（Gait Quality Rewards）

##### feet_air_time（滞空時間報酬）

**数学的定義:**
```python
# first_contact: 空中から接地した瞬間のフラグ
# feet_air_time: 各足の滞空時間（秒）
rew = Σ(feet_air_time - 0.5) * first_contact
rew *= (||commands_xy|| > 0.1)  # 移動コマンドがある場合のみ
```

**物理的意図:**
- 足を地面から離す時間を報酬化
- 引きずり歩き（shuffling）を防止
- 周期的な足の持ち上げを促進

**定量的解釈:**
- 滞空時間0.5秒がブレークイーブン
- 0.3秒滞空 → (0.3-0.5)×1.0 = -0.2（ペナルティ）
- 0.7秒滞空 → (0.7-0.5)×1.0 = +0.2（報酬）
- 歩行周波数1.5Hzでは1周期0.67秒、遊脚期0.3秒が典型的

**設計根拠:**
- 0.5秒オフセットは「十分な滞空時間」の閾値
- 移動コマンドがない場合は静止が望ましいため報酬を無効化

---

##### contact（接地フェーズ報酬）【Unitree G1/H1独自】

**数学的定義:**
```python
# leg_phase: 各脚の歩行位相 [0, 1)
# is_stance = leg_phase < 0.55（スタンス期）
# contact: 実際の接地状態（True/False）
for each leg:
    res += (~(contact XOR is_stance))  # XORの否定
```

**物理的意図:**
- 歩行フェーズと実際の接地状態の整合性を報酬化
- スタンス期（位相0-0.55）では接地、スイング期（0.55-1.0）では非接地を期待

**定量的解釈:**
- 両脚が期待通り → reward=2.0×0.2×dt
- 片脚のみ一致 → reward=1.0×0.2×dt
- 両脚とも不一致 → reward=0.0

**歩行パターンへの影響:**
```
位相0.0-0.55: 左脚スタンス（接地期待）、右脚スイング（非接地期待）
位相0.55-1.0: 左脚スイング、右脚スタンス
→ 自然な交互歩行パターンを誘導
```

**exp006問題への対応:**
- exp006の接地率0%問題を直接的に解決
- フェーズに同期した接地/非接地を報酬化することで「浮遊」を防止

---

##### feet_swing_height（遊脚高さペナルティ）【Unitree G1/H1独自】

**数学的定義:**
```python
# swing_height_target = 0.03m（BSL-Droid用）
pos_error = (feet_pos_z - target)²
penalty = Σ(pos_error * (~contact))  # 非接地の足のみ
```

**物理的意図:**
- 遊脚期の足を目標高さまで持ち上げることを促す
- つまずき防止、障害物回避能力の基礎

**定量的解釈:**
- 目標0.03mに対して足が地面スレスレ（0.01m）の場合:
  - error=(0.03-0.01)²=0.0004 → ペナルティ=0.0004×5.0×dt
- 足が高すぎる（0.06m）場合も同様にペナルティ

**BSL-Droid調整:**
- Unitree: target=0.08m, scale=-20.0
- V1: target=0.03m, scale=-5.0
- ロボットサイズ比（0.23/1.0）と緩和係数を考慮

---

##### contact_no_vel（接地時足速度ペナルティ）【Unitree H1独自】

**数学的定義:**
```python
vel_xy = feet_vel[:, :, :2]  # 水平方向速度のみ
vel_sq = Σ(vel_xy²)
penalty = Σ(vel_sq * contact)  # 接地している足のみ
```

**物理的意図:**
- 接地中の足が地面を滑ることを防止
- 安定した支持相を実現

**定量的解釈:**
- 接地足が0.1m/sで滑る場合: ペナルティ=0.01×0.1×dt
- 0.5m/sで滑る場合: ペナルティ=0.25×0.1×dt
- 小さいスケール(-0.1)は、完全な静止を強制しない設計

**設計根拠:**
- 歩行中は体重移動により多少の滑りは自然
- 過度な抑制は不自然な歩行や学習の不安定化を招く

---

##### hip_pos（股関節位置ペナルティ）【BSL-Droid追加】

**数学的定義:**
```python
hip_angles = [left_hip_yaw, left_hip_roll, right_hip_yaw, right_hip_roll]
penalty = Σ(hip_angles²)
```

**物理的意図:**
- hip_yaw（内外旋）とhip_roll（内外転）の過度な変位を抑制
- 脚が八の字に開く問題（exp006）を防止

**定量的解釈:**
- hip_roll 15°（0.26rad）の場合: 0.26²×0.5×dt
- hip_roll 30°（0.52rad）の場合: 0.52²×0.5×dt（4倍のペナルティ）

**exp006問題への対応:**
- exp006ではhip_rollが0〜91°まで変動
- このペナルティにより中立位置（0°）近傍を維持

---

#### 1.3.4 エネルギー効率ペナルティ（Energy Efficiency）

##### torques（トルクペナルティ）

**数学的定義:**
```python
penalty = Σ(actions²)  # 実際はアクション値の二乗和
```

**物理的意図:**
- 大きなトルク（モーター出力）を抑制
- エネルギー効率の良い歩行を促進

**定量的解釈:**
- スケール-1e-5は非常に小さい（微調整用）
- アクション値10の場合: 100×1e-5×dt = 0.001×dt
- 主報酬を阻害せず、同等の動作の中でエネルギー効率の良い方を選好

---

##### action_rate（アクション変化率ペナルティ）

**数学的定義:**
```python
penalty = Σ(actions - last_actions)²
```

**物理的意図:**
- アクションの急激な変化を抑制
- 滑らかな関節軌道を促進
- 高周波振動（震え）の防止

**定量的解釈:**
- スケール-0.01
- アクションが1ステップで0.5変化: 0.25×0.01×dt
- exp006の高周波振動問題への直接的対策

**設計根拠:**
- 実機では急激なアクション変化はモーターに負荷
- シミュレーションでも物理的に不自然な挙動を抑制

---

##### dof_acc（関節加速度ペナルティ）

**数学的定義:**
```python
dof_acc = (dof_vel - last_dof_vel) / dt
penalty = Σ(dof_acc²)
```

**物理的意図:**
- 関節の急激な加減速を抑制
- 滑らかな動作を促進

**定量的解釈:**
- スケール-2.5e-7は非常に小さい
- dt=0.02sで速度が1rad/s変化 → 加速度50rad/s²
  - ペナルティ=2500×2.5e-7×dt ≈ 0.0125×dt

**action_rateとの違い:**
- action_rateは指令値の変化、dof_accは実際の関節運動の変化
- 両者を併用することで、指令と結果の両方で滑らかさを促進

---

#### 1.3.5 報酬バランスの設計原則

Unitree RL Gymの報酬設計には以下の原則がある：

**1. 主報酬とペナルティのバランス**
```
主報酬（tracking_lin_vel + tracking_ang_vel + feet_air_time + contact + alive）
> ペナルティ合計
```
これにより、タスク達成へのインセンティブを維持しつつ、品質を向上させる。

**2. スケールの階層構造**
- O(1): 主タスク報酬（tracking_lin_vel）
- O(0.1-0.5): 副次的報酬/重要ペナルティ
- O(1e-4〜1e-7): 微調整用ペナルティ

**3. 報酬のdt乗算**
```python
self.reward_scales[name] *= self.dt  # 全報酬にdt=0.02を乗算
```
これにより、報酬の合計がシミュレーション時間に比例し、制御周期に依存しない学習が可能。

### 1.4 exp004（フルスクラッチ試行錯誤）との比較検討

#### 1.4.1 報酬設計パラダイムの変化

exp004では「Dense Reward Shaping」から「Minimalist」へ、そしてexp007では「Unitree参考実装」へと報酬設計パラダイムが変遷した。

| パラダイム | 実験 | 報酬項目数 | 特徴 |
|-----------|------|-----------|------|
| **Dense Shaping** | exp004 V9-V17 | 18-25項目 | 関節レベルの詳細制約、報酬ハッキングリスク高 |
| **Minimalist** | exp004 V18 | 6項目 | タスク報酬のみ、創発的学習を期待 |
| **Unitree参考** | exp007 V1 | 15項目 | 歩行フェーズ報酬による構造的誘導 |

#### 1.4.2 exp004の報酬設計の問題点

exp004 V9-V17で発生した問題とその原因：

| 問題 | exp004での報酬設計 | 結果 |
|------|-------------------|------|
| **報酬ハッキング** | `hip_pitch_alternation`で交互歩行を強制 | 両脚同期（+0.697）でも報酬最大化 |
| **報酬項目間の競合** | `forward_progress` vs `action_rate` | スケール調整の無限ループ |
| **創発的歩行の阻害** | 関節角度の詳細制約 | 不自然な歩行パターン |
| **接地検出なし** | `feet_air_time`未実装 | 浮遊状態（接地率0%） |

**exp004の教訓（exp004_reward_design_survey.mdより）**:
> 「報酬項目数の増加に反比例して歩行品質が悪化する現象が観察された」

#### 1.4.3 exp007（Unitree参考）で解決した問題

| exp004の問題 | exp007での解決策 | 根拠 |
|-------------|-----------------|------|
| **接地率0%** | `contact`報酬（フェーズ整合性） | Unitree G1/H1での成功実績 |
| **hip_roll異常（0〜91°）** | `hip_pos`ペナルティ | 股関節変位の二乗ペナルティ |
| **高周波振動** | `action_rate` + `dof_acc` | Unitree標準のスムーズネス報酬 |
| **報酬バランス崩壊** | Unitreeのスケール体系を参考 | 実績あるスケール値を出発点に |

#### 1.4.4 報酬構造の比較

**exp004 V19（11項目、Minimalist拡張）**:
```python
reward_scales = {

    "tracking_lin_vel": 2.0,
    "tracking_ang_vel": 0.5,
    "alive": 1.0,
    # 姿勢制御
    "orientation": -3.0,
    "base_height": -3.0,
    "pitch_penalty": -2.0,    # 独自追加
    # エネルギー効率
    "torques": -1e-4,
    "dof_acc": -1e-6,
    # 歩行パターン
    "feet_air_time": 1.0,
    "single_stance": 0.3,      # 独自追加
    "hip_pitch_velocity": 0.5, # 独自追加
}
```

#### 1.4.5 設計思想の根本的な違い

| 観点 | exp004（試行錯誤） | exp007（Unitree参考） |
|------|-------------------|---------------------|
| **交互歩行の誘導** | `hip_pitch_alternation`（関節レベル） | `contact`報酬（接地フェーズレベル） |
| **足上げの誘導** | 暗黙的（足先空間で試行） | `feet_swing_height`で明示的に目標高さ設定 |
| **接地状態の扱い** | 検出できず（座標ベース失敗） | `contact_threshold`でZ座標ベース判定 |
| **スケール設計** | 経験的調整（スケール調整の無限ループ） | Unitree実績値をベースに比率調整 |
| **報酬の階層構造** | なし（全報酬フラット） | O(1)〜O(1e-7)の明確な階層 |

#### 1.4.6 Unitree参考実装の優位性

**1. 歩行フェーズとの整合性報酬（`contact`）**

exp004では「どの関節をどう動かすか」を報酬で指定しようとしたが、Unitreeでは「いつ接地し、いつ足を上げるか」という高レベルな指針のみを与える。

```
exp004の問題: hip_pitch角度を直接制約 → 報酬ハッキング発生
exp007の解決: 歩行フェーズと接地状態の一致を報酬化 → 自然な交互歩行が創発
```

**2. 実績あるスケールバランス**

exp004では「主報酬 < ペナルティ合計」となり「何もしない」が最適解になる問題が発生した。Unitreeのスケール体系は実機で検証済みであり、このバランス問題が解決されている。

**3. 遊脚高さの明示的制御（`feet_swing_height`）**

exp004の足先空間アプローチでは「足を上げる」ことを直接報酬化できなかったが、Unitreeの`feet_swing_height`は目標高さを明示的に設定する。

#### 1.4.7 考察：なぜUnitree参考が有効か

**1. 産業レベルの実証**

Unitree G1/H1は商用製品であり、その報酬設計は数千時間の実機検証を経ている。exp004の試行錯誤で得られた知見（「Less is More」「Task-level > Joint-level」など）は、Unitreeの設計に既に反映されていた。

**2. 二脚特有の課題への対応**

Unitree G1/H1は二脚ヒューマノイドであり、四脚ロボット向けのLegged Gymよりも直接的に参考になる。特に以下の点：
- 両脚浮遊（Double Support失敗）への対応
- 遊脚高さ制御によるつまずき防止
- 股関節変位の抑制

**3. 報酬設計のモジュール性**

Unitreeの報酬関数は、それぞれが独立した物理的意味を持ち、スケール調整が容易である。exp004の「相互干渉する報酬項目」とは対照的に、Unitreeの報酬は「追加しても他を壊さない」設計になっている。

#### 1.4.8 残された課題

Unitree参考実装でも、BSL-Droid固有の課題への対応が必要：

| 課題 | Unitree設定 | BSL-Droid調整 | 理由 |
|------|------------|--------------|------|
| ロボットサイズ | G1: 1.2m | 0.19m | 脚長・高さ目標の比率調整 |
| 逆関節構造 | 通常膝 | 逆関節 | 膝角度の正負が逆 |
| 接地検出 | 接触力ベース | Z座標ベース | Genesis APIの制約 |

これらの調整は、V1の訓練結果を踏まえて反復的に改善していく。
