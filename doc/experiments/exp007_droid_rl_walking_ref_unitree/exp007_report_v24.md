# V24: 遊脚横方向速度ペナルティによる内股軌道の解消

## 概要

V24では、V23で観測された「遊脚の内股軌道」（体の内側を通るようにカーブを描いて前後移動する）を解消するため、遊脚時の横方向（Y軸）速度をペナルティ化する新規報酬関数`swing_foot_lateral_velocity`を追加する。

V23ではタップダンス（地面の2回蹴り）が解消されたものの、遊脚軌道が内股気味になっている課題が残っていた。これはエネルギー最小化によりhip rollが脱力気味になることで、重心安定化のために内側を通る軌道が選択されるためと推測される。

本バージョンでは、ユーザー提案に基づき、遊脚の速度成分を目標値と同じ軸方向（前進時は前後方向のみ）に揃える報酬を導入し、hip rollを適切に使って脚を真っ直ぐ前後に振る動機を与える。

## 前バージョンからの改善事項

### V23で観測された課題

- **内股軌道**: 遊脚が体の内側を通るようにカーブを描いて前後移動する
- **hip_roll範囲**: 左脚[-0.394, 0.000] rad、右脚[-0.004, 0.401] rad
  - 左脚は負の範囲（内側傾斜）、右脚は正の範囲（外側傾斜）を取る傾向
  - これは脚が体の内側を通る軌道になっている証拠

### 改善策（V24）

| パラメータ | V23値 | V24値 | 変更理由 |
|-----------|-------|-------|---------|
| swing_foot_lateral_velocity | - | -0.5 | 遊脚の横方向速度を抑制し内股を解消 |
| 報酬項目数 | 18 | 19 | +1項目 |

## 設計詳細

### 新規報酬関数: `_reward_swing_foot_lateral_velocity`

```python
def _reward_swing_foot_lateral_velocity(self):
    """遊脚の横方向速度ペナルティ（V24追加）

    【設計原理】
    V23課題: 遊脚が体の内側を通る軌道（内股）になっている
    原因: hip rollが脱力気味で、エネルギー最小化のために内側を通る軌道が選択される
    対策: 遊脚（非接地）時の速度指令と直交する速度成分をペナルティ化

    数学的には:
      cmd_dir = cmd / |cmd|  (指令方向の単位ベクトル)
      vel_parallel = vel · cmd_dir  (指令方向成分)
      |vel_perp|² = |vel|² - vel_parallel²  (直交成分の二乗)

    これにより、前進コマンド時はY軸成分が、横移動コマンド時はX軸成分が、
    斜め移動時は指令方向と直交する成分がペナルティ化される。
    前後左右自由自在に歩けるポリシーを学習できる。
    """
    # 速度指令ベクトル (cmd_x, cmd_y)
    cmd_x, cmd_y = self.commands[:, 0], self.commands[:, 1]
    cmd_norm = torch.sqrt(cmd_x**2 + cmd_y**2 + 1e-8)
    is_moving = cmd_norm > 0.05
    cmd_dir_x, cmd_dir_y = cmd_x / cmd_norm, cmd_y / cmd_norm

    # 遊脚速度ベクトルと直交成分の計算
    vel_x, vel_y = self.feet_vel[:, :, 0], self.feet_vel[:, :, 1]
    vel_sq = vel_x**2 + vel_y**2
    vel_parallel = vel_x * cmd_dir_x.unsqueeze(1) + vel_y * cmd_dir_y.unsqueeze(1)
    vel_perp_sq = torch.clamp(vel_sq - vel_parallel**2, min=0)

    # 遊脚時のみペナルティ
    swing_mask = (~contacts).float()
    penalty = torch.sum(vel_perp_sq * swing_mask, dim=1)
    return penalty * is_moving.float()
```

### contact_no_velとの違い

| 報酬項目 | 対象 | 速度成分 | 目的 |
|---------|------|---------|------|
| contact_no_vel | 接地中の足 | 水平方向（XY） | 足滑り防止 |
| swing_foot_lateral_velocity | 遊脚（非接地） | **指令方向と直交する成分** | 内股防止 |

**汎用性**: 前後左右自由自在に歩けるポリシーを学習できる。

### スケール設定の根拠

`swing_foot_lateral_velocity`のスケール **-0.5** の設定根拠:

1. `contact_no_vel`（-0.1）より強め: 内股軌道の直接原因に対処するため
2. `hip_pos`（-0.5）と同程度: 股関節制御に関連するため
3. 段階的調整の余地: 効果不足なら-1.0へ、強すぎれば-0.3へ調整可能

## 考察と改善案

### 成功点

1. **内股軌道の解消**: `swing_foot_lateral_velocity`ペナルティにより、hip_pos報酬が-0.0335から-0.0139に改善し、遊脚が体の内側を通る軌道が解消された。
2. **hip_roll範囲の改善**: 左脚[-0.248, 0.000]、右脚[0.000, 0.191]とV23より範囲が狭まり、より適切な脚の振り出しが実現。
3. **安定した歩行速度**: 目標0.15 m/sに対し0.204 m/sを達成し、速度追従は良好。
4. **良好な交互歩行**: hip_pitch相関-0.845で、左右脚の交互動作が維持されている。

### 課題

1. **タップダンス（2回蹴り）の再発**: ユーザーの目視評価で、地面を2回蹴る動作が周期的に発生。

2. **feet_air_time報酬の悪化**: V23の-0.0158からV24では**-0.0203に悪化**。これはV22（-0.0202）と同程度であり、タップダンス再発の直接的な原因。

3. **原因分析**:
   - `swing_foot_lateral_velocity`ペナルティ（-0.5）が遊脚の動きを過度に制限
   - 遊脚の横方向速度を抑制した結果、足を素早く下ろす傾向が生じた
   - 結果として滞空時間が短縮し、feet_air_time報酬が負のまま改善しなかった
   - V23で解消されたタップダンスが、新たなペナルティの副作用として再発

4. **片足接地率の低下**: 91.2%から88.2%に低下（タップダンスによる短い接地サイクルの影響）

### 次バージョンへの提案

**V25: swing_foot_lateral_velocityスケール緩和**

| パラメータ | V24値 | V25提案値 | 変更理由 |
|-----------|-------|----------|---------|
| swing_foot_lateral_velocity | -0.5 | **-0.2** | 内股解消効果を維持しつつ遊脚動作の自由度を確保 |

**根拠**:
1. V24で内股は解消されたため、-0.5ほどの強いペナルティは不要
2. -0.2は`contact_no_vel`（-0.1）の2倍程度で、適度な抑制効果を維持
3. 遊脚の自然な振り動作を許容し、適切な滞空時間を確保
4. 報酬項目数は19項目のまま維持（新規追加なし）

**期待効果**:
- feet_air_time報酬が正に転換（V23レベルへ回復）
- タップダンスの解消
- 内股軌道の再発防止（緩やかなペナルティは維持）

**代替案**（V25で効果不十分な場合）:
- 案B: feet_air_time報酬スケールを1.0→1.5に強化
- 案C: air_time_offsetを0.2→0.15に短縮
- 最終手段: swing_foot_lateral_velocityを削除しV23設定に回帰

## まとめ

V24では、V23の内股軌道課題に対処するため、遊脚時の横方向速度をペナルティ化する報酬関数`swing_foot_lateral_velocity`（スケール-0.5）を追加した。

**結果**:
- ✅ 内股軌道は解消された（hip_pos報酬: -0.0335 → -0.0139）
- ❌ タップダンス（2回蹴り）が再発（feet_air_time報酬: -0.0158 → -0.0203）

**原因**: `swing_foot_lateral_velocity`ペナルティが遊脚の動きを過度に制限し、滞空時間が短縮した。

**次バージョン（V25）への提案**: `swing_foot_lateral_velocity`スケールを-0.5から-0.2に緩和し、内股解消効果を維持しつつタップダンスを解消することを目指す。

## 備考

- 本改善案はユーザー提案（「遊脚の速度成分が目標値と同じ軸の成分だけになるようにする」）に基づく
- 報酬項目数が19項目となり、ルールの推奨範囲（15-17項目）をわずかに超えるが、18-20項目は「注意」レベルであり許容範囲内
- 効果が不十分な場合はスケールを-1.0に強化、強すぎる場合は-0.3に緩和することを検討
