# EXP007: 実験の進め方とルール

本ドキュメントは、exp007（BSL-Droid Simplified二脚ロボットのUnitree RL Gym参考実装による強化学習歩容獲得実験）を進める上での手順、コマンド、スクリプト、ルールを記載する。

## 1. バージョン管理とコード信頼性

強化学習環境とトレーニングスクリプトのバージョン管理において、以下の原則を厳守する（exp002から継承）:

### 1.1 統一環境クラス原則（重要）

**環境クラスファイルは1個のみ**とし、新しい報酬関数や機能は既存の統一ファイルに累積させる。

```
rl_ws/biped_walking/envs/droid_env_unitree.py  # 唯一の環境クラスファイル
```

**禁止事項**:
- ❌ バージョンごとに環境クラスファイルを作成する（例: `droid_env_unitree_v1.py`, `v2.py`, `v3.py`）
- ❌ ファイル数が爆発的に増加する構成

**正しい運用**:
- ✅ 全ての報酬関数を `droid_env_unitree.py` に実装
- ✅ トレーニングスクリプト側の `reward_scales` で報酬の有効/無効を制御
- ✅ `reward_scales` に含めない、または値を0に設定すれば報酬は無効化される

### 1.2 報酬関数の命名規則

環境クラスに新しい報酬関数を追加する際は、以下の命名規則に従うこと。

**メソッド名**:
```python
def _reward_{name}(self):
    """報酬関数の説明（追加バージョンを明記）

    【設計原理】
    - 報酬の目的と設計意図を記述

    【参考文献】（任意）
    - 参考にした論文やリポジトリがあれば記載
    """
    # 実装
    return reward_tensor
```

**命名規約**:
- プレフィックス: `_reward_`（必須）
- 名前部分: snake_case（例: `tracking_lin_vel`, `feet_air_time`, `single_foot_contact`）
- 戻り値: `torch.Tensor` shape `(num_envs,)`

**reward_scalesでの参照**:
```python
reward_scales = {
    "{name}": 1.0,  # _reward_{name} が呼び出される
}
```

**docstringに含めるべき情報**:
1. 報酬の目的（1行）
2. 追加バージョン（例: V7追加）
3. 設計原理（なぜこの報酬が必要か）
4. 参考文献（任意）

### 1.3 トレーニングスクリプトのコピー原則

新バージョン（V(n+1)）を作成する際は、**トレーニングスクリプトのみ**を完全コピーしてから編集を開始する。

```bash
# 例: V1からV2を作成する場合
cd rl_ws/biped_walking
cp train/droid_train_unitree_v1.py train/droid_train_unitree_v2.py
# 注意: envファイルはコピーしない（統一環境クラスを使用）
```

### 1.4 最小変更原則

コピー後は、改善目的に必要な箇所**のみ**を編集する:
- docstring・報酬スケール（`reward_scales`）・コマンド設定など、変更が必要な部分だけを特定
- 既存の動作検証済みコードは可能な限り保持
- 不必要な再実装や構造変更を避ける

### 1.5 段階的検証

各バージョンで以下を確認:
- 構文エラーがないこと（実行前に検証）
- 依存パラメータ（command_cfg等）の整合性
- 初期化処理の完全性（qpos次元数等）

### 1.6 理由

- ゼロから再実装すると、細かい設定ミス（qpos次元、command_cfg、feet_indices初期化等）が混入しやすい
- 既存の動作確認済みコードをベースにすることで、新規追加部分のデバッグに集中できる
- 信頼性の高い反復的改善が可能
- 環境クラスを統一することで、ファイル数の爆発を防ぎ、保守性を維持できる

## 2. 実験記録のルールと責務の分離

本実験における記録ファイルの責務分離は、`AGENTS.md` の「実験記録の責務分離」ルールに準拠する。

### 2.1 ファイル責務一覧

| ファイル | 責務 |
|----------|------|
| `exp007_droid_rl_walking_ref_unitree.md` | 実験概要・背景・ロボット仕様・バージョン一覧（インデックス） |
| `exp007_report_v{N}.md` | 各バージョンの設計・実装・結果（**1バージョン = 1ファイル**） |
| `exp007_rules.md` | 本ドキュメント（実験ルール・手順・コマンド） |
| `exp007_unitree_rl_gym_survey.md` | Unitree RL Gymの報酬設計に関する調査**のみ** |

### 2.2 バージョン別レポート原則（重要）

**1バージョン = 1ファイル**とし、各バージョンの実験結果は個別ファイルに記録する。

**命名規則**:
```
exp007_report_v{N}.md  # 例: exp007_report_v1.md, exp007_report_v2.md, exp007_report_v6.md
```

**禁止事項**:
- ❌ 複数バージョンの結果を1ファイルに累積する
- ❌ 主レポート（`exp007_droid_rl_walking_ref_unitree.md`）に詳細な実験結果を記載する

**正しい運用**:
- ✅ 新バージョン開始時に `exp007_v{N}.md` を新規作成
- ✅ 主レポートにはバージョン一覧とリンクのみを記載
- ✅ 各バージョンファイルには設計意図・パラメータ・結果・考察を自己完結的に記載

**理由**:
- 各バージョンのレポートが自己完結的になり、参照しやすい
- ファイルサイズの肥大化を防止
- 過去バージョンの結果が新バージョン追加時に影響を受けない

### 2.3 失敗実験の記録方針

**失敗も重要な知見である**。以下の方針で失敗実験を記録すること。

**記録すべき「失敗」の例**:
- 静止ポリシーへの収束（ロボットが動かない）
- 転倒の多発（エピソード長が短い）
- 意図と逆の動作（後退、旋回など）
- 振動・痙攣の発生
- 特定の関節の異常動作

**失敗実験のレポート記載事項**:
1. **何が起きたか**: 観察された問題を客観的に記述
2. **なぜ起きたか**: 原因の仮説を記述
3. **何を学んだか**: 次バージョンへの教訓を記述

**禁止事項**:
- ❌ 失敗実験のレポートを省略する
- ❌ 「失敗したので詳細は省略」と書く
- ❌ 失敗バージョンを主レポートのバージョン一覧から除外する

**理由**:
- 同じ失敗を繰り返さないため
- 何が効果的で何が効果的でないかを把握するため
- 実験の完全な履歴を維持するため

### 2.5 新バージョン作成時の手順（Coding Agent向け）【厳守】

新バージョン（V(n+1)）の実装を依頼された際、Coding Agentは以下のチェックリストを**この順番で**実行すること。

#### チェックリスト

- [ ] **1. 前バージョンのレポートを参照して精査する**
  - [ ] `exp007_report_v{N-1}.md` を読み込む
  - [ ] 「次バージョンへの提案」セクションの内容を確認する
  - [ ] 課題と改善策を把握する

- [ ] **2. 必要に応じて追加評価を実行する**（任意）
  - [ ] 前バージョンのレポートに不足情報がある場合のみ
  - [ ] 評価: `uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{N-1} --no-viewer --duration 10`
  - [ ] ログ分析: `uv run python scripts/analyze_training_log.py droid-walking-unitree-v{N-1}`

- [ ] **3. 改善策に基づいてトレーニングスクリプトを作成する**
  - [ ] 1.3 トレーニングスクリプトのコピー原則に従う
  - [ ] 必要に応じて環境クラスに新しい報酬関数を追加する（1.1, 1.2 に従う）
  - [ ] 構文チェック: `uv run python -m py_compile biped_walking/train/droid_train_unitree_v{N}.py`
  - [ ] lintチェック: `uv run ruff check biped_walking/`（エラーがあれば `--fix` で自動修正）
  - [ ] 型チェック: `uv run mypy biped_walking/`（コード品質ルールは `AGENTS.md` 参照）

- [ ] **4. 作業中バージョンのレポートを作成する**
  - [ ] `exp007_report_v{N}.md` を新規作成する
  - [ ] 2.4 のテンプレートに従って記述する
  - [ ] 「結果」セクションは訓練完了後に記入する旨を明記する
  - [ ] 実装したトレーニングスクリプトに関する事項（設計詳細、パラメータ変更）を詳細に記載する

- [ ] **5. 主レポートにバージョン参照を追加する**
  - [ ] `exp007_droid_rl_walking_ref_unitree.md` のバージョン一覧表に新バージョンを追加する
  - [ ] バージョン進化の概要図も更新する

- [ ] **6. ユーザに学習コマンドを提示する**
  - [ ] 学習の実行はユーザが行う
  - [ ] **自動で学習を開始してはならない**

#### 禁止事項

- ❌ 前バージョンのレポートを確認せずにスクリプトを作成する
- ❌ レポートを作成せずにスクリプトのみを提供する
- ❌ 主レポートへの参照追加を忘れる
- ❌ Coding Agentが自動で学習を開始する

#### 理由

- 前バージョンの知見を確実に引き継ぐため
- 実験の追跡可能性を確保するため
- ユーザが学習開始のタイミングを制御できるようにするため

## 3. 主要コマンド

### 3.1 トレーニング実行

```bash
cd rl_ws
uv run python biped_walking/train/droid_train_unitree_v{VERSION}.py --max_iterations 500
```

オプション:
- `--num_envs`: 並列環境数（デフォルト: 4096）
- `--max_iterations`: 最大イテレーション数（デフォルト: 500）
- `-e`, `--exp_name`: 実験名（デフォルト: `droid-walking-unitree-v{VERSION}`）

### 3.2 ポリシー評価（GUI付き）

```bash
cd rl_ws
uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{VERSION}
```

オプション:
- `--ckpt`: チェックポイント番号（デフォルト: 最新）
- `--duration`: 評価時間（秒）

### 3.3 ポリシー評価（ヘッドレス、定量評価用）

```bash
cd rl_ws
uv run python biped_walking/biped_eval.py -e droid-walking-unitree-v{VERSION} --no-viewer --duration 10
```

このコマンドは以下の定量指標を出力する:
- X/Y移動距離、平均速度
- 姿勢（Roll/Pitch/Yaw）
- hip_pitch相関（交互歩行の指標）
- DOF range sum（動きの大きさ）
- 接地パターン

このコマンドは実行されて結果がコンソールに現れるまでに読み込み時間がかかるものであるため，あなたが焦って再実行することを禁じる．

### 3.4 TensorBoard監視

```bash
cd rl_ws
uv run python -m tensorboard.main --logdir logs
# ブラウザで http://localhost:6006 を開く
```

## 5. 報酬設計原則

exp007ではUnitree RL Gymの報酬設計を参考にする。

詳細は [exp007_unitree_rl_gym_survey.md](exp007_unitree_rl_gym_survey.md) を参照。

### 5.1 報酬項目追加に関する厳格ルール（V10教訓に基づく）

**V1-V10の試行錯誤から得られた重要な教訓**を今後の実験で必ず遵守すること。

#### 報酬項目の追加は最終手段

```
❌ 問題発生 → 新しい報酬項目を追加
✅ 問題発生 → 既存項目の重み調整 → 効果なし → 項目の削除を検討 → 最後に追加を検討
```

**根拠**：
- V3-V4（16-17項目）で0.15-0.19 m/sを達成
- V7-V10（22項目）で不安定化、最終的に静止ポリシー（0.006 m/s）
- Unitree G1/H1は13-14項目で実機歩行を達成
- **項目を増やすほど良くなるわけではない。むしろ項目間の相互作用が予測困難になる。**

#### 新規項目追加前の必須チェックリスト

新しい報酬項目を追加する前に、以下を**全て**確認すること：

- [ ] **既存項目の重み調整で対応できないか？**
- [ ] **不要な項目を削除して簡素化できないか？**
- [ ] **Unitree RL Gymに同等の項目が存在するか？**（存在しない場合は慎重に）
- [ ] **V3-V4（動作成功時）に存在した項目か？**
- [ ] **追加による副作用（ペナルティ累積、相互干渉）を十分に検討したか？**
- [ ] **一度に追加するのは1項目のみか？**

#### 避けるべきパターン（実験で実証済み）

| パターン | 問題 | 実例 |
|---------|------|------|
| 複数項目の同時追加/強化 | 相互作用が予測困難 | V10: 5項目同時変更 → 静止ポリシー |
| 対称性報酬（symmetry）の強化 | hip_pitch同期誘発 | V9: symmetry 1.0 → 相関+0.605に悪化 |
| ペナルティの累積強化 | 動くこと自体がペナルティに | V7→V10でペナルティ倍増 → 静止 |
| 複雑な交互歩行報酬 | 意図しない効果 | alternating_gait/hip_pitch_antiphase → 効果なし |

#### 推奨するアプローチ

1. **シンプルさを維持**: 15-17項目を上限とする
2. **実績ベース**: V3-V4、Unitreeの設定を参照
3. **1変更1検証**: 一度に変更するのは1-2項目まで
4. **削除優先**: 追加より削除を優先的に検討
5. **ペナルティ控えめ**: 主報酬を上回らないよう設定

#### 問題別の対処指針

| 問題 | 推奨アプローチ | 避けるべきアプローチ |
|------|--------------|-------------------|
| 交互歩行が不十分 | single_foot_contactの重み調整 | symmetry/alternating_gait追加 |
| 足上げ不足 | swing_height_target調整 | feet_swing_height強化 |
| Yawドリフト | tracking_ang_vel調整 | 新規Yawペナルティ追加 |
| 動作が滑らかでない | action_rate微調整 | 複数ペナルティ強化 |
| 静止ポリシー | ペナルティ緩和、velocity_deficit確認 | 新規報酬追加 |

### 5.2 報酬項目数の目安

| 項目数 | 評価 | 根拠 |
|-------|------|------|
| 13-14 | ✅ Unitree推奨 | G1/H1で実機歩行達成 |
| 15-17 | ✅ BSL-Droid推奨 | V3-V4で0.15-0.19 m/s達成 |
| 18-20 | ⚠️ 注意 | 相互作用リスク増加 |
| 21以上 | ❌ 非推奨 | V7-V10で不安定化実証済み |

### 5.3 階層的報酬設計の原則

報酬スケールは以下の階層構造を遵守すること：

1. **主報酬**（最優先）：tracking_lin_vel（1.0-1.5）> tracking_ang_vel（0.5）
2. **歩行品質報酬**（次優先）：feet_air_time, contact, single_foot_contact
3. **補助報酬**（低優先）：alive, その他
4. **ペナルティ**（抑制のみ）：**絶対値の合計が主報酬を上回らないこと**

**重要**: ペナルティの累積が主報酬を上回ると、「動かない」が最適解になる（V10で実証）。

## 6. トラブルシューティング

### 6.1 「動かない」ポリシーが学習された（静止ポリシー問題）

**これはV1-V10で繰り返し発生した最も深刻な問題である。**

#### 原因候補（優先度順）

1. **ペナルティの累積効果**（最も可能性が高い）
   - V10で実証: 5項目のペナルティを同時強化 → 静止ポリシー
   - 動くこと自体が大きなペナルティを受ける状態

2. **報酬項目の相互干渉**
   - symmetry報酬がhip_pitch同期を誘発（V9で実証）
   - 複雑な報酬設計が意図しない効果を生む

3. **alive報酬が相対的に高い**
   - Unitreeの0.15は大きすぎる場合がある
   - 動かずに生存するだけで報酬を獲得

4. **速度目標が低すぎる**
   - 低速目標では動く動機が弱い
   - tracking_lin_velの勾配が緩くなる

#### 対処（優先度順）

1. **ペナルティを緩和する**（最優先）
   - V7レベルに戻す: ang_vel_xy=-0.05, orientation=-0.5
   - 複数項目を一度に強化した場合は全て戻す

2. **不要な報酬項目を削除する**
   - symmetry, alternating_gait, feet_stumble等を削除
   - 15-17項目に簡素化

3. **velocity_deficitを確認**
   - 速度未達ペナルティが有効か確認
   - V6で効果実証済み: -2.0

4. **動作していたバージョンに回帰**
   - V3-V4またはV7の設定に戻す
   - そこから最小限の変更で再試行

#### チェックポイント

```
報酬最大化 ≠ 歩行品質最大化
```

V10は最終報酬98.12（V7の73.44より高い）だが、X速度は0.006 m/s（静止）。
報酬値ではなく、**実際の動作**で評価すること。

### 6.2 接地フェーズ検出がうまくいかない

原因候補:
1. contact_threshold（接触力閾値）が不適切
2. 足先リンクのインデックスが誤っている

対処:
1. 接触力の値をログ出力して確認
2. feet_indicesの設定を確認

### 6.3 feet_swing_heightペナルティが強すぎる

原因候補:
- Unitreeのスケール（-20.0）がBSL-Droidには過剰

対処:
- スケールを1/10程度（-2.0）から開始し、調整
