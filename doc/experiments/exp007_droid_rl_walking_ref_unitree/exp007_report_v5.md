## Phase 5: V5 - 足首振動抑制と対称性改善

### 5.1 V5設計概要

V4の「足首振動」「左右非対称動作」「交互歩行未達成」問題に対して、先行研究サーベイ（[exp007_unitree_rl_gym_survey.md](exp007_unitree_rl_gym_survey.md) セクション7.3.3, 7.4.3）に基づく対策を実装。

#### 5.1.1 V4の成功と課題（再掲）

**成功点**:
- 接地検出の修正成功（single foot contact 98.4%）
- single_foot_contact報酬の機能（0.6901）
- Yawドリフトの改善（+1.77°）
- 関節可動域の増加（hip_pitch 10-13°）

**残存問題**:

| 問題 | V4での症状 | 原因分析 |
|------|-----------|---------|
| 足首高速振動 | ankle_roll velocity std: 2.7-3.6 rad/s | dof_vel=-0.001が不十分 |
| 左右非対称 | hip_roll可動域: L=0.321, R=0.435 rad | 対称性報酬がない |
| 交互歩行未達成 | hip_pitch相関: +0.792 | 足首振動で速度達成可能 |
| feet_air_time負 | 報酬値: -0.0275 | offset=0.5秒が長すぎる |

#### 5.1.2 V5の設計方針

V4の課題分析に基づき、以下の3つの改善戦略を採用:

1. **低速度戦略**: 足首振動に頼らない速度域で交互歩行を学習
2. **振動抑制強化**: dof_velペナルティを5倍に強化
3. **対称性誘導**: symmetry報酬で左右バランスを報酬化
4. **feet_air_time正常化**: offset=0.25秒でBSL-Droidサイズに適合

#### 5.1.3 V5パラメータ変更一覧

| パラメータ | V4値 | V5値 | 変更理由 |
|-----------|------|------|---------|
| **lin_vel_x_range** | [0.2, 0.3] m/s | **[0.10, 0.15] m/s** | 低速で交互歩行獲得（振動依存排除） |
| **gait_frequency** | 1.0 Hz | **0.8 Hz** | 遊脚期延長で交互歩行促進 |
| **dof_vel** | -0.001 | **-0.005** | 関節速度ペナルティ5倍強化 |
| **air_time_offset** | 0.5 s | **0.25 s** | 小型ロボット用に調整（新規パラメータ） |
| **symmetry** | - | **0.3（新規）** | 左右対称性報酬の導入 |

**注意**: 目標速度低下はV2で「静止ポリシー」を誘発したが、V5では以下の対策が機能する見込み:
- `single_foot_contact=0.8`（接地検出が機能済み）
- `velocity_deficit=-0.5`（速度未達ペナルティ）
- `alive=0.03`（控えめな生存報酬）

### 5.2 新規機能の実装

#### 5.2.1 air_time_offsetの設定可能化

V4まではオフセット値0.5秒がハードコードされていたが、V5では`reward_cfg`から設定可能に変更。

**droid_env_unitree.pyへの追加**:
```python
# feet_air_time報酬のオフセット（V5追加）
# デフォルト0.5秒はUnitree G1/H1向け、小型ロボットは0.25秒推奨
self.air_time_offset = self.reward_cfg.get("air_time_offset", 0.5)
```

**_reward_feet_air_timeの修正**:
```python
def _reward_feet_air_time(self):
    """滞空時間報酬（Unitree方式）

    オフセット値はreward_cfg["air_time_offset"]で設定可能（デフォルト: 0.5秒）。
    小型ロボット（BSL-Droid）では0.25秒程度が適切。
    """
    if self.feet_air_time is None or self._first_contact is None:
        return torch.zeros(self.num_envs, dtype=gs.tc_float, device=gs.device)

    # air_time_offsetは設定可能（V5追加）
    rew_air_time = torch.sum(
        (self.feet_air_time - self.air_time_offset) * self._first_contact.float(), dim=1
    )
    rew_air_time *= (torch.norm(self.commands[:, :2], dim=1) > 0.1).float()
    return rew_air_time
```

**offset設定の理論的根拠**:
- 歩行周期 = 1/0.8Hz = 1.25秒
- 遊脚期（片足）= 1.25秒 × 0.45 ≈ 0.56秒
- offset=0.25秒であれば、正常な遊脚で約0.31秒の正味報酬を獲得可能
- V4のoffset=0.5秒では、0.56-0.5=0.06秒とほぼゼロまたは負になっていた

#### 5.2.2 symmetry報酬（左右対称性報酬）

**設計意図**:
- hip_pitchは交互歩行で逆位相が理想なので除外
- hip_roll, knee, ankleは同位相（対称）が理想

**droid_env_unitree.pyへの追加**:
```python
def _reward_symmetry(self):
    """左右脚対称性報酬（V5追加）

    hip_roll, knee, ankleの左右差をペナルティ化することで、
    左右非対称な動きを抑制する。

    【参考文献】
    Leveraging Symmetry in RL-based Legged Locomotion Control (IROS 2024)
    """
    # 左脚: roll(1), knee(3), ankle(4)
    # 右脚: roll(6), knee(8), ankle(9)
    left_roll = self.dof_pos[:, self.left_hip_roll_idx]
    right_roll = self.dof_pos[:, self.right_hip_roll_idx]
    left_knee = self.dof_pos[:, self.left_knee_idx]
    right_knee = self.dof_pos[:, self.right_knee_idx]
    left_ankle = self.dof_pos[:, self.left_ankle_idx]
    right_ankle = self.dof_pos[:, self.right_ankle_idx]

    # 左右差の二乗和
    symmetry_error = (
        torch.square(left_roll - (-right_roll)) +  # rollは符号反転で対称
        torch.square(left_knee - right_knee) +
        torch.square(left_ankle - right_ankle)
    )

    # ガウシアン型報酬（誤差が小さいほど報酬が高い）
    return torch.exp(-symmetry_error / 0.5)
```

**対称性の定義**:
- `hip_roll`: 左+α, 右-αが対称（左外側傾き = 右内側傾き）
- `knee`, `ankle`: 左右同値が対称

### 5.9 考察とV6への展望

#### 5.9.1 V5失敗の根本原因分析

**V5はV2と同じ「静止ポリシー」に収束した**。原因を詳細に分析する。

**1. ガウシアン報酬関数と低目標速度の組み合わせ問題**

`tracking_lin_vel`報酬の計算式:
```python
reward = exp(-error² / tracking_sigma²)
# tracking_sigma = 0.25
```

| 条件 | 目標速度 | 実速度 | error | 報酬 |
|------|---------|-------|-------|------|
| V4（動作） | 0.25 m/s | 0.244 m/s | 0.006 | **0.999** |
| V5（静止） | 0.125 m/s | 0.003 m/s | 0.122 | **0.787** |

**問題**: V5は速度が98%低下しても、報酬は21%しか低下しない。
ガウシアン報酬は「ほぼ達成」と「完全未達」を十分に区別できない。

**2. velocity_deficitペナルティの不十分さ**

```
velocity_deficit = (target - actual)² × scale
V5: (0.125 - 0.003)² × (-0.5) = -0.0074
```

このペナルティ（-0.0074）は`tracking_lin_vel`報酬（0.8243）の**1%未満**であり、
静止を回避するインセンティブとしては弱すぎた。

**3. V2との比較で見える教訓**

| 条件 | V2 | V5 | 結果 |
|------|-----|-----|------|
| 目標速度 | 0.10-0.15 | 0.10-0.15 | 同じ（静止誘発） |
| alive報酬 | 0.15 | 0.03 | V5が低い |
| single_foot_contact | 未実装 | 0.8（機能中） | V5が有利 |
| velocity_deficit | 未実装 | -0.5（弱い） | V5にあるが不十分 |
| **結果** | 静止 | 静止 | **目標速度が支配的** |

**結論**: 目標速度の低下が静止ポリシーの**支配的な原因**である。
V5で追加した対策（dof_vel強化、symmetry、air_time_offset）は静止を回避するには不十分だった。

#### 5.9.3 V6設計方針：低速度・ゆっくり歩容の維持

**ユーザー要件**: BSL-Droidのサイズ（身長約35cm）に適した低速度（0.10-0.15 m/s）とゆっくりした歩容周期（0.8 Hz）を維持したい。

**戦略**: 目標速度・歩容周波数はV5を維持しつつ、ガウシアン報酬関数を鋭化することで静止ポリシーを回避する。

#### 5.9.4 V6の核心：tracking_sigma調整による静止識別

**問題の本質**:

V5の静止ポリシー問題は、ガウシアン報酬関数（`tracking_sigma=0.25`）が低目標速度において「静止」と「歩行」を十分に識別できないことに起因する。

```python
# tracking_lin_vel報酬の計算式
reward = exp(-error² / tracking_sigma²)
```

**sigma=0.25（V5）での報酬識別能力**:

| 状態 | 目標速度 | 実速度 | error | 報酬 | 備考 |
|------|---------|-------|-------|------|------|
| 歩行達成 | 0.125 m/s | 0.125 m/s | 0.000 | **1.000** | 100% |
| 静止 | 0.125 m/s | 0.003 m/s | 0.122 | **0.787** | 79%（21%低下のみ） |

**問題**: 速度が**98%低下**しても報酬は**21%しか低下しない**。
この差では静止を回避するインセンティブとして不十分。

**sigma=0.10（V6案）での報酬識別能力**:

| 状態 | 目標速度 | 実速度 | error | 報酬 | 備考 |
|------|---------|-------|-------|------|------|

#### 5.9.7 リスク分析

**リスク1: tracking_sigma=0.10が厳しすぎる可能性**

sigma=0.10では、目標速度から±0.1 m/sずれると報酬が急激に低下する。
低速度（0.10-0.15 m/s）では、わずかな速度変動でも報酬が不安定になる可能性がある。

**対策**:
- 訓練初期の報酬変動を監視
- 必要に応じてsigma=0.15を検討（中間値）

**リスク2: velocity_deficit=-2.0が強すぎる可能性**

静止時のペナルティが大きすぎると、転倒リスクを冒してでも動こうとする可能性。

**対策**:
- エピソード長の推移を監視
- 転倒増加が見られたら-1.5や-1.0に調整

> **セクション7.5**: 静止ポリシー問題の対策
> 「報酬関数の設計で解決できない場合、カリキュラム学習が有効」

V6ではまず報酬関数（tracking_sigma）の調整で解決を試みる。
これが失敗した場合、V7以降でカリキュラム学習（高速度→低速度への段階的移行）を検討する。
